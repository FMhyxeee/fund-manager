# 06-边界与接口清单

本文档明确 `fund-manager` 和 `OpenClaw` 的职责边界，也明确仓库内部 domain kernel、typed contract 和 runtime bridge 的边界。

目标不是抽象地说“一个管业务，一个管 Agent”，而是把常见动作和模块归属说清楚。

## 1. 一句话原则

- `fund-manager` 负责投资领域里的真相、规则、留痕和 typed contract。
- `OpenClaw` 负责模型编排、渠道交互、外部工具和通知。

进一步说：

- 只要涉及 canonical 投资数据、确定性计算、审计记录、领域写路径，就归 `fund-manager`。
- 只要涉及模型选择、聊天上下文、渠道发送、提醒通知、跨项目工具编排，就归 `OpenClaw`。

## 2. `fund-manager` 内部的三层边界

即使都在同一个仓库里，也要分清三层：

### 2.1 Domain kernel

拥有：

- canonical facts
- deterministic decisions
- research signals
- append-only audit records

主要模块：

- `core/domain`
- `core/services`
- `core/watchlist`
- `storage`

### 2.2 Typed contract layer

拥有：

- AI workflow 输入 contract
- AI workflow 输出 contract

主要模块：

- `core/fact_packs.py`
- `core/ai_artifacts.py`

### 2.3 Runtime bridge layer

拥有：

- runtime protocol
- prompt metadata
- manual runtime adapter

主要模块：

- `agents/runtime/contracts.py`
- `agents/runtime/shared.py`
- `agents/runtime/*_agent.py`

这层不能拥有：

- canonical truth
- persisted artifact schema
- prompt 中隐藏的 business logic

## 3. `fund-manager` 拥有什么

### Canonical truth

- `portfolio`
- `fund_master`
- `transaction`
- `position_lot`
- `nav_snapshot`
- `portfolio_snapshot`

### Deterministic decision truth

- active policy
- `decision_run`
- `decision_feedback`
- `decision_transaction_link`

### Deterministic research signals

- watchlist candidates
- style leaders
- candidate fit

### Append-only narrative artifacts

- `review_report`
- `strategy_proposal`
- `agent_debate_log`

### Typed interfaces

- API
- CLI
- MCP
- internal typed tools
- fact packs
- AI artifacts

## 4. `OpenClaw` 拥有什么

- agent runtime
- model routing
- auth profile
- tool permission
- prompt orchestration
- subagent routing
- chat / Feishu / email / summary delivery
- cron / heartbeat / retry policy
- 外部搜索和跨项目通用工具

## 5. 动作级归属

### 必须归 `fund-manager`

- 导入持仓
- 导入交易
- 同步基金资料和 NAV
- 保存组合快照
- 运行 deterministic daily decision
- 记录人工 feedback
- 做 transaction reconciliation
- 生成 deterministic watchlist / style leader / fit result
- 保存 report / proposal / debate log

### 必须归 `OpenClaw`

- 决定什么时候触发哪个 workflow
- 选择哪个模型、哪个 provider、哪个 profile
- 把结果发给谁、用什么语气发
- 从自然语言中理解“执行了 / 跳过 / 延后”
- 混合外部新闻搜索和渠道上下文做摘要

## 6. 允许的调用方式

`OpenClaw` 可以：

- 调用 `fund-manager` API
- 调用 `fund-manager` CLI
- 调用 `fund-manager` MCP
- 读取 `fund-manager` 的结构化输出后做摘要或通知

`OpenClaw` 不可以：

- 直接写 `fund-manager` 数据库
- 在 prompt 里重算 portfolio truth
- 因为某个 AI 建议存在，就默认推断为已执行交易

## 7. 典型端到端流程

### 流程 A：Daily Decision

1. `OpenClaw` 决定何时触发。
2. `fund-manager` 读取快照与 policy，生成 `decision_run`。
3. `OpenClaw` 读取结构化结果，决定如何向人类展示。

边界：

- 决策真相归 `fund-manager`
- 编排和表达归 `OpenClaw`

### 流程 B：Weekly Review

1. `fund-manager` 组装 `WeeklyReviewFacts`。
2. runtime 依据 protocol 返回 `ReviewAgentOutput`。
3. `fund-manager` 持久化 `review_report`。
4. `OpenClaw` 决定是否、何时、如何通知。

边界：

- facts 和 artifact 的持久化归 `fund-manager`
- 渠道交互归 `OpenClaw`

### 流程 C：Monthly Strategy Debate

1. `fund-manager` 组装 `StrategyDebateFacts`。
2. Strategy / Challenger / Judge 运行。
3. `fund-manager` 保存 `strategy_proposal` 和 debate log。
4. `OpenClaw` 可以读取结果并二次总结。

边界：

- strategy artifact 的 append-only 留痕归 `fund-manager`
- 外部增强和渠道化表达归 `OpenClaw`

### 流程 D：Research Signal

1. `fund-manager` 用 deterministic rule 生成 watchlist / leader / fit。
2. 结果可进入 API / CLI / MCP / workflow context。
3. AI 可以引用这些 signal，但不能把它们写回 canonical truth。

## 8. 新功能放哪边的判定题

1. 这个功能会不会改 canonical 投资数据或新增审计记录？
   - 会，放 `fund-manager`
2. 这个功能能不能在没有聊天上下文、没有特定模型、没有渠道配置的情况下正确运行？
   - 能，优先放 `fund-manager`
3. 这个功能是不是主要在解决“用什么模型、什么时候跑、发给谁、怎么说”？
   - 是，放 `OpenClaw`
4. 这个功能是不是 AI workflow 的 typed input / output contract？
   - 是，放 `core/fact_packs.py` 或 `core/ai_artifacts.py`
5. 这个功能是不是 runtime protocol、prompt metadata 或 manual adapter？
   - 是，放 `agents/runtime`

## 9. 明确禁止的反模式

- `OpenClaw` 直接连库写 `fund-manager` 表
- prompt 内复制一份 portfolio/accounting 逻辑
- runtime 实现文件偷偷拥有 canonical DTO
- runtime helper 模块偷偷拥有 persisted artifact schema
- 研究信号被直接当成 policy truth
- AI narrative 被直接当成 canonical truth

## 10. 当前推荐的对接方式

短期内，推荐按下面顺序对接：

1. `fund-manager` API
2. `fund-manager` CLI
3. `fund-manager` MCP
4. internal typed tools

无论 transport 用哪一个，边界规则不变：

- 投资领域真相只在 `fund-manager`
- typed contract 由 `fund-manager` 定义
- runtime 编排和渠道交互只在 `OpenClaw`
