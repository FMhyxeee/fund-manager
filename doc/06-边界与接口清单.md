# 06-边界与接口清单

这份文档用来明确 `fund-manager` 和 `OpenClaw` 的职责边界。

目标不是抽象地说“一个管业务，一个管 Agent”，而是把常见动作拆开，明确谁拥有真相、谁负责编排、谁不能越界。

## 1. 一句话原则

- `fund-manager` 负责投资领域里的真相、规则、留痕。
- `OpenClaw` 负责 Agent 层的编排、交互、通知。

进一步说：

- 只要涉及 canonical 投资数据、确定性计算、审计记录、领域写路径，就归 `fund-manager`。
- 只要涉及模型选择、对话上下文、渠道发送、提醒通知、跨项目工具编排，就归 `OpenClaw`。

## 2. 谁拥有什么

### `fund-manager` 拥有

- canonical 数据模型和数据库真相
  - `portfolio`
  - `fund_master`
  - `transaction`
  - `position_lot`
  - `nav_snapshot`
  - `portfolio_snapshot`
  - `review_report`
  - `strategy_proposal`
  - `decision_run`
  - `decision_feedback`
  - `decision_transaction_link`
- 确定性投资计算
  - 估值
  - 权重
  - 成本
  - 收益率
  - 回撤
  - policy band 检查
  - suggested rebalance amount
  - feedback 和 transaction 的 reconciliation
- 基金领域数据接入
  - AKShare 同步
  - holdings CSV 导入
  - transactions CSV 导入
  - fund search
  - NAV 刷新
- 投资领域审计产物
  - 快照
  - 报告
  - 策略提案
  - 日决策
  - 手工反馈
  - 对账链接

### `OpenClaw` 拥有

- agent runtime
  - 模型路由
  - auth profile
  - tool 权限
  - subagent 编排
- operator-facing 交互
  - chat 对话
  - Feishu/Discord/邮件等渠道发送
  - 结果摘要
  - 提醒与提醒文案
- orchestration
  - cron
  - heartbeat
  - retry policy
  - 什么时候跑哪个 workflow
  - 跑完要不要通知、通知给谁
- 非基金专属的通用能力
  - web search
  - mail
  - calendar
  - shell
  - GitHub

## 3. 动作级归属清单

下面这些动作，默认应该归 `fund-manager`。

### 读 canonical 投资状态

- 获取组合列表
- 获取组合快照
- 获取持仓明细
- 获取组合估值历史
- 获取组合指标摘要
- 获取基金资料
- 获取基金净值历史

这些动作应该通过 `fund-manager` 的 API、CLI、MCP 或 typed tool 暴露。

### 写 canonical 投资状态

- 导入初始持仓
- 导入交易流水
- 同步当前持仓基金数据
- 刷新净值快照
- 保存组合快照
- 创建或更新 policy 的 append-only 版本
- 运行 daily decision 并落 `decision_run`
- 记录人工 feedback 并落 `decision_feedback`
- 将导入交易回挂到 prior feedback
- 运行 weekly review 并落 `review_report`
- 运行 monthly strategy debate 并落 `strategy_proposal`

只要动作会改 canonical 投资数据或生成投资审计记录，就不能放在 `OpenClaw` prompt 或 channel handler 里。

下面这些动作，默认应该归 `OpenClaw`。

### 编排和交互

- 每天几点跑 `daily_snapshot`
- 每天几点跑 `daily_decision`
- 跑完以后是否发飞书
- 飞书里发详细版还是简版
- 如果有动作建议，要不要追问用户“今天是否已执行”
- 用户在聊天里回复“执行了/跳过/延后”以后，是否自动调用 `fund-manager` 写 feedback
- 失败以后什么时候重试
- 多 agent 讨论时用哪个模型、哪个 profile

### 对外信息增强

- 把 `fund-manager` 的 deterministic 输出和财经新闻拼在一起做解释
- 基于渠道上下文调整语气和摘要长度
- 在 group chat 里决定回复、沉默还是加 reaction

## 4. 允许的接口边界

`OpenClaw` 可以做这些事：

- 调用 `fund-manager` API
- 调用 `fund-manager` CLI
- 调用 `fund-manager` MCP
- 调用 `fund-manager` agent tool
- 读取 `fund-manager` 已经落好的结构化结果并进行解释、摘要、通知

`OpenClaw` 不可以做这些事：

- 直接写 `fund-manager` 数据库表
- 在 prompt 里重算持仓、收益、权重、policy 命中结果
- 把聊天上下文当成 canonical 交易事实
- 因为 agent 给过建议，就默认视为“已经交易”

`fund-manager` 可以做这些事：

- 维护自身领域工作流的内部实现
- 输出结构化 artifact 供 `OpenClaw` 调用
- 提供 deterministic domain service 和 repo

`fund-manager` 不应该做这些事：

- 依赖 `OpenClaw` 的 profile、memory、channel 或 cron 才能正确运行
- 内建 Feishu/Discord 发送逻辑作为核心正确性的一部分
- 管理用户聊天记忆、消息格式偏好、group chat 行为策略

## 5. 典型端到端流程

### 流程 A: 日决策

1. `OpenClaw` 的 cron/heartbeat 判断该运行 `daily_decision`。
2. `OpenClaw` 调用 `fund-manager` 的 workflow 接口。
3. `fund-manager` 用 canonical 数据和 active policy 生成 `decision_run`。
4. `OpenClaw` 读取结构化结果，转成适合渠道的摘要并发送。

这里的边界是：

- 决策内容和 action 列表属于 `fund-manager`
- 什么时候跑、怎么说、发到哪属于 `OpenClaw`

### 流程 B: 人工执行反馈

1. `OpenClaw` 把 `decision_run` 发给用户。
2. 用户在聊天里回复“执行了”“跳过”“延期”。
3. `OpenClaw` 负责理解这句自然语言，并定位到具体 `decision_run` + `action_index`。
4. `OpenClaw` 调 `fund-manager` 的 feedback 写接口。
5. `fund-manager` 记录 `decision_feedback`，必要时尝试 reconciliation。

这里的边界是：

- 自然语言理解和交互属于 `OpenClaw`
- feedback 落库和 reconciliation 属于 `fund-manager`

### 流程 C: 导入交易后自动对账

1. 用户手工完成基金交易。
2. 交易 CSV 被导入 `fund-manager`。
3. `fund-manager` 规范化交易、写入 `transaction`，并尝试和最近的 executed feedback 建链。
4. `OpenClaw` 可以再读取结果，告诉用户哪些动作已经成功回挂，哪些还没匹配上。

这里的边界是：

- 导入、标准化、对账属于 `fund-manager`
- 提醒和解释属于 `OpenClaw`

### 流程 D: 周报和月度讨论

1. `OpenClaw` 或用户显式触发 workflow。
2. `fund-manager` 生成 deterministic facts。
3. 如果需要 agent 讨论，agent 运行可以在 `fund-manager` 内部 workflow 完成，也可以由 `OpenClaw` 编排后调用 `fund-manager` 落结果。
4. 最终保存的 `review_report` / `strategy_proposal` 仍然属于 `fund-manager`。

这里的边界是：

- 作为投资审计产物的最终落库归 `fund-manager`
- 对外呈现、通知和会话编排归 `OpenClaw`

## 6. 新功能放哪边的判定题

遇到新需求，按下面顺序判断：

1. 这个功能会不会改 canonical 投资数据，或者新增投资审计记录？
   - 会，放 `fund-manager`
2. 这个功能能不能在没有聊天上下文、没有特定模型、没有渠道配置的情况下独立正确运行？
   - 能，优先放 `fund-manager`
3. 这个功能是不是主要在解决“什么时候跑、怎么讲、发给谁、用哪个 agent”？
   - 是，放 `OpenClaw`
4. 这个功能是不是别的项目也会复用，不是基金场景专属？
   - 是，优先放 `OpenClaw`

## 7. 明确禁止的反模式

- `OpenClaw` 直接连库写 `fund-manager` 表
- prompt 内复制一份 portfolio/accounting 逻辑
- `fund-manager` 保存 Feishu 消息模板、频道 routing 规则、agent profile 配置
- `OpenClaw` 根据“上次建议过买入”就自己假定已经成交
- `fund-manager` 把 agent runtime 成功与否当成投资真相的一部分

## 8. 当前推荐的对接方式

短期内，推荐按下面的优先顺序对接：

1. `fund-manager` CLI
2. `fund-manager` API
3. `fund-manager` MCP
4. `fund-manager` typed agent tools

无论 transport 用哪一个，边界规则不变：

- 投资领域真相只在 `fund-manager`
- Agent 壳层交互和编排只在 `OpenClaw`

接口落地顺序见 [`07-接口补齐计划`](./07-%E6%8E%A5%E5%8F%A3%E8%A1%A5%E9%BD%90%E8%AE%A1%E5%88%92.md)。
