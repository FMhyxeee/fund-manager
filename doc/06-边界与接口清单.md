# 06-边界与接口清单

本文档描述当前精简版 `fund-manager` 与 `OpenClaw` 的边界，以及仓库内部接口的使用边界。

当前版本已经移除 AI workflow、MCP、scheduler、admin CLI、dashboard、report、policy/decision persistence 等产品面。`fund-manager` 现在只保留基金账本、组合读模型和观察池核心。

## 1. 一句话原则

- `fund-manager` 负责基金领域事实、确定性计算、账本写入和观察池状态。
- `OpenClaw` 负责模型编排、聊天上下文、渠道交互、提醒通知和跨项目工具。

只要涉及 canonical 投资数据或领域写路径，就走 `fund-manager` 的受控接口。只要涉及“什么时候说、怎么说、发给谁、用哪个模型”，就归 `OpenClaw`。

## 2. `fund-manager` 当前拥有的真相

Canonical tables:

- `fund_master`
- `portfolio`
- `transaction`
- `position_lot`
- `nav_snapshot`
- `watchlist_item`

Core behaviors:

- `transaction` 是权威交易账本。
- `position_lot` 是从交易账本确定性重建的物化持仓状态。
- 组合 snapshot / positions / metrics / valuation history 是读模型，不再依赖旧的 `portfolio_snapshot` 表。
- `watchlist_item` 是观察状态，不是持仓、决策或交易指令。

## 3. 当前公开接口

主入口是 FastAPI，base path 为 `/api/v1`。

允许公开的 route:

- `GET /api/v1/health`
- `GET /api/v1/portfolios`
- `GET /api/v1/portfolios/{portfolio_id}/snapshot`
- `GET /api/v1/portfolios/{portfolio_id}/positions`
- `GET /api/v1/portfolios/{portfolio_id}/metrics`
- `GET /api/v1/portfolios/{portfolio_id}/valuation-history`
- `GET /api/v1/funds/{fund_code}`
- `GET /api/v1/funds/{fund_code}/nav-history`
- `GET /api/v1/transactions`
- `GET /api/v1/transactions/{transaction_id}`
- `POST /api/v1/transactions`
- `GET /api/v1/watchlist`
- `POST /api/v1/watchlist/items`
- `DELETE /api/v1/watchlist/items/{fund_code}`

详细调用方式见 [`../skills/fund-manager-interfaces/references/core-api.md`](../skills/fund-manager-interfaces/references/core-api.md)。

## 4. 仓库内部接口边界

### API layer

位置：`src/fund_manager/apps/api`

职责：

- request / response contract
- HTTP error translation
- dependency injection
- 调用 service 层

不应承担：

- 重算持仓
- 直接写账本和持仓
- 持有业务规则

### Service layer

位置：

- `src/fund_manager/core/services`
- `src/fund_manager/core/watchlist`

职责：

- 交易追加和 lot 重建
- 组合读模型组装
- 观察池 add/list/remove
- 领域校验和 DTO 输出

关键服务：

- `PortfolioService`
- `TransactionService`
- `TransactionLotSyncService`
- `FundWatchlistService`

### Storage layer

位置：`src/fund_manager/storage`

职责：

- SQLAlchemy model
- migration
- repository

Repositories 是 service 的实现细节。外部调用者和 OpenClaw 不应依赖 repository 或直接写数据库。

## 5. `OpenClaw` 当前允许做什么

允许：

- 调用 `fund-manager` FastAPI
- 读取结构化 API 响应后做摘要或通知
- 在人类明确确认交易已执行后，调用 `POST /api/v1/transactions`
- 对数据库做只读诊断查询
- 在明确 repair/migration 任务中执行受控 SQL

不允许：

- 直接写 `transaction`、`position_lot`、`watchlist_item`
- 在 prompt 里重算 portfolio truth
- 因为观察池、新闻或 AI 建议存在，就推断交易已经执行
- 调用已删除的 MCP、scheduler、admin CLI、workflow、report、policy/decision route

| 读取组合 snapshot | `fund-manager` | `GET /api/v1/portfolios/{id}/snapshot` |
| 读取持仓 | `fund-manager` | `GET /api/v1/portfolios/{id}/positions` |
| 读取组合指标 | `fund-manager` | `GET /api/v1/portfolios/{id}/metrics` |
| 读取基金资料 | `fund-manager` | `GET /api/v1/funds/{code}` |

| 选择模型或 agent | `OpenClaw` | OpenClaw runtime |

## 7. 典型流程

### 组合读取

1. 调用 `/portfolios` 找到目标组合。
2. 调用 `/snapshot`、`/positions` 或 `/metrics`。
3. 如果有 `missing_nav_fund_codes`，展示为数据缺口，不要补零。

### 人工确认后的交易追加

1. 人类明确表示交易已经实际执行。
2. OpenClaw 整理 `portfolio_id` 或 `portfolio_name`、`fund_code`、`fund_name`、`trade_date`、`trade_type`、`units`、`gross_amount` 等字段。
3. 调用 `POST /api/v1/transactions`。
4. 以响应里的 `transaction` 和 `lot_sync` 作为账本写入和 lot 重建成功的依据。

### 观察池管理

1. 用 `POST /watchlist/items` 添加或重新激活观察基金。
2. 用 `GET /watchlist` 读取 active list。
3. 用 `DELETE /watchlist/items/{fund_code}` 软删除。
4. 不把观察池当成交易指令。

## 8. 明确禁止的反模式

- route handler 里直接实现持仓计算或账本写入
- 外部 agent 直接写数据库
- 为了兼容旧流程重新暴露 `/decisions`、`/reports`、`/workflows`
- 让 watchlist 污染 canonical accounting truth
- 将缺失 NAV 视为 0
- 在没有 migration 的情况下改 schema
