适用对象：

## 4. 核心模块职责

## 4.1 `core/domain`

定义领域对象与纯逻辑。

建议对象：
- `Fund`
- `Transaction`
- `PositionLot`
- `PortfolioSnapshot`
- `StrategyProposal`

要求：
- 尽量无副作用
- 不直接依赖 ORM
- 可单测

---

## 4.2 `core/services`

封装业务逻辑与计算逻辑。

建议服务：
- `PortfolioService`
- `PricingService`
- `AnalyticsService`
- `StrategySupportService`

### PortfolioService
职责：
- 根据交易与净值构建当前组合状态
- 汇总组合持仓

### PricingService
职责：
- 获取和刷新基金净值
- 对接数据适配器

### AnalyticsService
职责：
- 计算收益
- 计算回撤
- 计算仓位与集中度
- 生成指标摘要

### StrategySupportService
职责：
- 检查是否偏离目标配置
- 生成再平衡候选
- 输出结构化策略辅助数据

---

## 4.3 `core/rules`

定义可配置规则，不混在 Prompt 中。

建议规则：
- `risk_rules.py`
- `rebalance_rules.py`

### 示例规则
- 单只基金权重上限
- 组合最大回撤警戒值
- 单周下跌超过阈值时进入观察名单
- 偏离目标配置超过阈值时输出再平衡提醒

---

## 4.4 `data_adapters`

负责外部数据获取与清洗。

建议模块：
- `akshare_adapter.py`
- `import_holdings.py`
- `import_transactions.py`
- `normalizer.py`

### `akshare_adapter.py` v1 能力
- `search_fund(fund_code_or_name)`
- `get_fund_profile(fund_code)`
- `get_fund_nav_history(fund_code, start_date, end_date)`

### `import_holdings.py`
负责：
- 读取 CSV / Excel
- 校验字段
- 标准化为内部结构
- 输出可入库记录

### `import_transactions.py`
负责：
- 导入买入 / 卖出 / 分红 / 转换流水
- 标准化交易类型

---

## 4.5 `storage`

负责数据库连接、ORM 模型与仓储层。

### `storage/models.py`
定义 ORM 表结构。

### `storage/repo/`
定义仓储对象，例如：
- `FundRepo`
- `PortfolioRepo`
- `TransactionRepo`
- `NavRepo`
- `ReportRepo`

要求：
- repo 层不承担复杂业务决策
- repo 只负责读写与查询封装

这是 Agent 能看到的唯一业务入口。

工具必须：

- 输出结构化

建议工具：
- `get_portfolio_snapshot`
- `get_position_breakdown`
- `get_fund_profile`
- `get_fund_nav_history`
- `compute_drawdown`
- `detect_risk_flags`
- `generate_rebalance_candidates`
- `save_review_report`
- `save_strategy_proposal`
- `save_debate_log`

## 6. 输入文件规范

## 6.1 初始持仓导入 CSV

推荐格式：

```csv
fund_code,fund_name,units,avg_cost,total_cost,portfolio_name
000001,示例基金A,1234.56,1.2034,1485.20,main
000002,示例基金B,345.00,2.1060,726.57,main
```

字段要求：
- `fund_code` 必填
- `units` 必填，必须为正数
- `avg_cost` 与 `total_cost` 至少二选一
- `portfolio_name` 默认 `main`

## 6.2 交易流水导入 CSV

推荐格式：

```csv
fund_code,trade_date,trade_type,units,amount,fee,nav_at_trade,source,note
000001,2026-03-01,buy,100.0,120.35,0.0,1.2035,manual,首次导入
000001,2026-03-10,dividend,5.2,0.0,0.0,,manual,现金分红转份额
```

`trade_type` 枚举建议：
- `buy`
- `sell`
- `dividend`
- `convert_in`
- `convert_out`
- `adjust`

## 9. Weekly Loop 技术流程

推荐伪代码：

```python
async def run_weekly_review(portfolio_id: int, period_start, period_end):
    run_id = create_run_id("weekly_review")
    log_event("weekly_review_started", {"run_id": run_id, "portfolio_id": portfolio_id})

    refresh_nav_data(portfolio_id, period_end)
    rebuild_positions(portfolio_id, period_end)
    metrics = compute_portfolio_metrics(portfolio_id, period_start, period_end)
    risk_flags = detect_risk_flags(portfolio_id, period_end)

    facts = data_agent(metrics=metrics, risk_flags=risk_flags)
    review = review_agent(facts=facts, metrics=metrics)
    strategy = strategy_agent(review=review, metrics=metrics, risk_flags=risk_flags)
    challenge = challenger_agent(strategy=strategy, metrics=metrics, risk_flags=risk_flags)
    decision = judge_agent(
        facts=facts,
        review=review,
        strategy=strategy,
        challenge=challenge,
    )

    report = render_weekly_markdown(
        metrics=metrics,
        review=review,
        strategy=strategy,
        challenge=challenge,
        decision=decision,
    )

    save_review_report(...)
    save_strategy_proposal(...)
    save_debate_log(...)
    log_event("weekly_review_finished", {"run_id": run_id})

    return report
```

## 10. AGENTS.md 建议初稿

根目录 `AGENTS.md` 建议至少包含以下内容：

```md
# Project
This repository is a personal fund portfolio review and strategy assistant.
It is a decision-support system, not an auto-trading system.

# Core constraints
- Never let LLMs compute authoritative portfolio numbers directly.
- All authoritative calculations must come from deterministic Python services.
- Agents may summarize, compare, challenge, and explain.
- Agents must not directly mutate accounting tables.
- All strategy outputs must include evidence-backed reasoning.

# Architecture rules
- Keep domain logic separate from adapters.
- Keep adapters separate from prompts.
- Keep agent prompts separate from business rules.
- Historical reports are append-only.

# Data integrity
- Schema changes require Alembic migrations.
- Core accounting logic changes require tests.
- Import pipelines must validate and normalize data before storage.

# Risk policy
- No auto-execution of trades in v1.
- No unsupported factual claims in reports.
- Weekly and monthly strategy proposals must be reviewed by ChallengerAgent.

# Coding style
- Use typed Python.
- Prefer small composable services.
- Write pure functions for calculations where possible.
- Avoid hidden side effects.

# Tests
- Unit tests are mandatory for metrics and transaction handling.
- Integration tests are mandatory for import pipelines and weekly loop.
```

## 12. 错误处理要求

### 数据层错误
- 导入失败必须给出具体字段错误
- 不允许静默丢弃记录

### Agent 层错误
- 任一 Agent 失败必须写入 `system_event_log`
- Weekly / Monthly loop 可以降级，但不能假装成功

### 外部接口错误
- 数据源异常要重试
- 超过阈值则记录失败并继续下一步可恢复流程

## 16. 总结

本技术文档的落地原则是：

- 先把账本做对
- 再把 loop 跑通
- 再把多 Agent 做稳
- 最后再追求界面与高级自动化

这会让项目更容易启动，也更容易长期演化。
