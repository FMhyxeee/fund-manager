# 基金投资管理器高阶方案（High-Level Solution）v1

## 1. 文档目标

本文档定义项目的**高阶架构、模块边界、运行机制、Agent 分工、数据流与部署策略**。

适用对象：
- Codex
- 系统设计者
- 未来要扩展本系统的任何工程代理

本文档回答的问题：
- 系统整体怎么拆
- 每一层负责什么
- Agent 如何协作
- loop 怎么运行
- 为什么选择这种架构

---

## 2. 架构总览

系统采用“五层架构”：

1. **数据接入层**
2. **事实与计算层**
3. **Agent 工具层**
4. **Agent 编排与 loop 层**
5. **展示与报告层**

整体目标：
- 保持账本与分析分离
- 保持工具与模型分离
- 保持运行时与开发时分离

---

## 3. 核心角色划分

### 3.1 开发时角色

#### Codex
负责：
- 生成代码
- 修改架构
- 添加测试
- 维护文档
- 协助构建 Agent 工作流

#### AGENTS.md
负责：
- 固化工程规范
- 限制 Codex 的行为边界
- 说明本项目的业务规则与风险规则

### 3.2 运行时角色

#### OpenClaw
负责：
- Agent runtime
- 调度 loop
- session / memory / routing
- 多模型分发
- 运行中的 Agent 执行管理

#### Fund Engine
负责：
- 账本事实
- 指标计算
- 风险规则
- 可验证的策略约束

#### 数据适配器
负责：
- 基金资料获取
- 净值抓取
- 数据清洗与标准化

## 4. 为什么采用这种架构

### 4.1 账本内核不能和 Agent 混在一起

如果把持仓计算、收益计算、回撤计算直接混入 Agent Prompt 或 tool-free 对话中，会导致：
- 结果不可验证
- 出错难定位
- 测试困难
- 历史一致性差

因此：
- 账本和指标必须由确定性引擎负责
- Agent 仅使用工具调用这些结果

### 4.2 订阅登录模式适合个人自托管

当前目标是：
- 主要给自己用
- 优先复用现有订阅能力
- 不优先构建正式面向外部用户的平台

因此架构重点是：
- 低部署复杂度
- 个人可信边界
- 后台 loop 可持续运行

### 4.3 多 Agent 只有在职责明确时才有价值

如果多个模型只是一起“发表观点”，会变成高成本、低信噪比的群聊。

因此设计采用：
- 事实代理
- 复盘代理
- 策略代理
- 反驳代理
- 裁决代理

---

## 5. 高阶模块设计

## 5.1 数据接入层

职责：
- 导入初始持仓
- 导入交易流水
- 获取基金公开资料
- 获取基金净值历史
- 做字段标准化

模块建议：
- `import_holdings`
- `import_transactions`
- `akshare_adapter`
- `fund_data_normalizer`

输入来源：
- 你的 CSV / Excel
- 公开基金数据源

输出结果：
- 统一的结构化实体
- 可入库的清洗结果

---

## 5.2 事实与计算层

职责：
- 维护组合事实状态
- 基于交易与净值重建持仓
- 计算收益、回撤、权重等核心指标
- 提供策略计算的约束边界

推荐拆分：
- `domain/portfolio`
- `domain/transaction`
- `domain/fund`
- `services/portfolio_service`
- `services/analytics_service`
- `rules/risk_rules`
- `rules/rebalance_rules`

原则：
- 不依赖 Prompt
- 可单元测试
- 可回放

---

## 5.3 Agent 工具层

职责：
- 把领域服务包装成 Agent 可调用工具
- 限制 Agent 访问方式
- 让 Agent 读取事实，而非自由操作底层存储

设计原则：
- 工具返回结构化数据
- 每个工具有明确输入 / 输出
- 工具允许读，谨慎写
- 写操作仅限“保存报告 / 保存建议 / 保存日志”

工具类别：
- 组合工具
- 基金工具
- 指标工具
- 风险工具
- 报告工具

---

## 5.4 Agent 编排层

职责：
- 管理多 Agent 协作
- 管理 loop 生命周期
- 管理 run id、trace、日志
- 管理跨模型 handoff

推荐采用以下工作流：

1. `CoordinatorAgent` 启动本轮任务
2. `DataAgent` 生产事实摘要
3. `ReviewAgent` 解释近期表现
4. `StrategyAgent` 生成策略草案
5. `ChallengerAgent` 提出反对意见
6. `JudgeAgent` 形成最终决策结论
7. `CoordinatorAgent` 保存结果

---

## 5.5 展示与报告层

职责：
- 展示组合状态
- 展示基金详情
- 展示报告与策略建议
- 展示历史 debate 与策略效果

v1 可以采用轻量化界面：
- FastAPI + 简单 HTML
- 或 Streamlit
- 或后续再接 Next.js

优先级：
- 先有报告
- 再有页面

---

## 6. 多 Agent 方案

## 6.1 Agent 列表

### DataAgent
输入：
- 组合状态
- 基金资料
- 净值历史

输出：
- 数据事实摘要
- 异常提示

### ReviewAgent
输入：
- 数据事实摘要
- 组合指标

输出：
- 近期表现解释
- 主要得失归因
- 观察项

### StrategyAgent
输入：
- 复盘结果
- 风险约束
- 组合目标

输出：
- 下一阶段策略草案
- 操作建议
- 置信度

### ChallengerAgent
输入：
- StrategyAgent 草案
- 风险标志
- 历史表现

输出：
- 反对意见
- 潜在偏差
- 风险警告

### JudgeAgent
输入：
- 所有前置 Agent 输出

输出：
- 最终结论
- 执行建议等级
- 待观察事项

---

## 6.2 模型分工原则

建议：
- 主模型负责编排、裁决、复杂推理
- 第二模型负责挑战与不同视角
- 必要时增加一个专门做总结的模型

关键原则：
- 模型不是随意分配
- 每个模型要绑定角色
- 同一轮 workflow 中保留模型名称与输出记录

---

## 7. Loop 设计

## 7.1 Loop 目标

Loop 不是无限对话，而是：

**定时触发 → 组装上下文 → 调用 Agent → 保存结果 → 结束本轮运行**

每次 loop 都是一个独立的可回放 run。

---

## 7.2 Daily Loop

目标：
- 更新净值
- 刷新组合快照
- 生成简短日报

流程：
1. 更新基金净值
2. 计算日度组合变化
3. 检查异常波动
4. 调用 DataAgent
5. 调用 ReviewAgent
6. 生成日报
7. 保存报告与日志

---

## 7.3 Weekly Loop

目标：
- 形成完整的周度复盘与策略建议

流程：
1. 汇总近 7 天数据
2. 生成组合指标
3. DataAgent 输出事实
4. ReviewAgent 输出复盘
5. StrategyAgent 输出策略草案
6. ChallengerAgent 输出反驳
7. JudgeAgent 给出最终结论
8. 保存周报、策略建议、debate log

---

## 7.4 Monthly Loop

目标：
- 面向长期配置与再平衡的月度决策

流程：
1. 计算月度收益与回撤
2. 计算目标配置偏离
3. 调用风险规则与再平衡候选
4. 执行多 Agent 辩论
5. 形成月报和再平衡建议
6. 保存结果

---

## 8. 记忆设计

## 8.1 长期记忆

内容包括：
- 风险偏好
- 长期目标
- 不做的策略
- 投资原则
- 偏好的分析风格

不包括：
- 原始账本明细
- 高频变动数据
- 重复的净值序列

## 8.2 运行记忆

每次 loop 只保留本次运行必要上下文：
- run id
- 输入摘要
- 中间产物
- 输出结果

---

## 9. 存储策略

建议拆分为三类存储：

### 9.1 主业务库
存：
- 基金资料
- 持仓
- 交易
- 净值
- 组合快照
- 报告
- 策略建议

### 9.2 日志库 / 日志表
存：
- loop 运行日志
- agent 输出摘要
- 工具调用记录
- 错误信息

### 9.3 文件输出
存：
- markdown 报告
- 导出结果
- 调试快照

---

## 10. 部署方案

## 10.1 v1 推荐部署形态

- Windows 主机
- WSL2 环境
- OpenClaw 以 daemon 形式运行
- Python 后端服务本地运行
- SQLite 本地存储

这样做的好处：
- 部署简单
- 便于本地调试
- 适合个人使用

## 10.2 后续升级形态

后续可升级为：
- Postgres 替代 SQLite
- 独立 API 服务
- 前端独立部署
- 容器化部署

## 12. 架构约束

1. Agent 不直接写底层账本数据
2. 指标计算必须从领域服务获取
3. 所有可执行建议必须带风险说明
4. 每次 loop 必须可追踪
5. 所有模型输出都必须落日志
6. 所有历史报告 append-only
