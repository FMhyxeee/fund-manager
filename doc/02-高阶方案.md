# 基金投资管理器高阶方案（High-Level Solution）

> 当前状态（2026-04-18）：本文保留为历史架构方案。当前仓库已经移除 AI workflow、policy/decision persistence、MCP、scheduler、admin CLI、report/dashboard 等产品面，只保留精简核心接口。当前可执行边界和接口以 `AGENTS.md`、`doc/06-边界与接口清单.md`、`doc/07-接口补齐计划.md` 和 `skills/fund-manager-interfaces/` 为准。

## 文档定位

本文档描述 `fund-manager` 在“投资领域内核 + 可替换 AI 分析层”模型下的目标架构、模块职责和关键数据流。

它关注的是：
- 层次如何拆
- 什么属于 deterministic domain
- 什么属于 research signal
- 什么属于 AI narrative
- `fund-manager` 与 `OpenClaw` 的连接契约该长什么样

## 总体架构

推荐架构不是“AI 驱动中心 + 数据支撑”，而是“领域内核在中心，AI 在外围”。

可以把系统分成 6 层：

1. 数据接入层
2. Canonical Truth 层
3. Deterministic Decision 层
4. Research Signal 层
5. AI Narrative Artifact 层
6. Transport / Orchestration 层

## 1. 数据接入层

职责：
- 导入 holdings CSV
- 导入 transactions CSV
- 同步基金资料
- 同步净值历史
- 对外部字段做归一化

对应模块：
- `data_adapters/import_holdings.py`
- `data_adapters/import_transactions.py`
- `data_adapters/akshare_adapter.py`
- 未来的其他 external adapters

约束：
- 只能负责抓取、校验、归一化
- 不负责定义 canonical accounting rules
- 不负责生成 narrative

## 2. Canonical Truth 层

这是整个系统的核心。

职责：
- 维护投资领域的 source of truth
- 保存 canonical tables
- 组装组合快照
- 计算权威 metrics

包括：
- `storage/models.py`
- `storage/repo/*.py`
- `core/domain/*.py`
- `core/services/portfolio_service.py`
- `core/services/analytics_service.py`
- `core/services/transaction_lot_sync_service.py`

该层输出的是：
- `PortfolioSnapshotDTO`
- positions
- valuation history
- deterministic metrics

该层不应该输出：
- AI 结论
- prompt 逻辑
- 渠道文案

## 3. Deterministic Decision 层

这是 canonical truth 之上的规则引擎层。

职责：
- 读取 active policy
- 检查 policy band
- 形成 deterministic daily decision
- 记录和回放可执行动作
- 接受人工反馈并做 reconciliation

包括：
- `core/services/policy_service.py`
- `core/services/decision_service.py`
- `core/services/decision_feedback_service.py`
- `core/services/decision_reconciliation_service.py`

持久化对象：
- `portfolio_policy`
- `decision_run`
- `decision_feedback`
- `decision_transaction_link`

关键边界：
- 这层可以生成“确定性动作建议”
- 但这层不等于“自动执行”
- 人工反馈才是 execution truth

## 4. Research Signal 层

这层用于承载“研究型读模型”。

职责：
- 生成 watchlist candidates
- 生成 style leaders
- 生成 candidate vs portfolio fit
- 未来承载 market radar / signal snapshot

当前对应模块：
- `core/watchlist/service.py`

推荐理解：
- 它是 deterministic read-model intelligence
- 它不是 canonical accounting truth
- 它不是 policy truth
- 它不是 auto-trading signal engine

这层产物应该被视为：
- 结构化研究信号
- 可解释的观察结果
- 可进入 review / strategy workflow 的 context

而不是：
- 直接下单指令
- 自动改写 policy 的依据

## 5. AI Narrative Artifact 层

这一层承接 AI 的位置，但 AI 不能反过来定义内核真相。

职责：
- 基于 deterministic facts 生成 narrative
- 输出 review / strategy / challenge / judgment
- 以 append-only artifact 形式持久化

当前对应模块：
- `core/fact_packs.py`
- `core/ai_artifacts.py`
- `agents/workflows/weekly_review.py`
- `agents/workflows/strategy_debate.py`
- `reports/exporters/markdown_exporter.py`

runtime bridge 的当前拆分：
- `agents/runtime/contracts.py`
  - runtime protocol
- `agents/runtime/shared.py`
  - prompt metadata loading and tiny shared helpers
- `agents/runtime/review_agent.py`
- `agents/runtime/strategy_agent.py`
- `agents/runtime/challenger_agent.py`
- `agents/runtime/judge_agent.py`
  - manual runtime implementations

推荐目标：
- `fund-manager` 内只保留最小可测试的 AI artifact workflow
- 生产环境中的 model routing、provider 选择、外部检索与多 agent 编排，逐步放到 `OpenClaw`

### 5.1 Fact Pack 原则

AI 不应直接读数据库表。

AI 应只接收结构化 Fact Pack，其中包含：
- `run_id`
- `portfolio_id`
- 时间范围
- deterministic metrics
- 缺失数据说明
- provenance / assumptions

当前代码落点：
- `WeeklyReviewFacts`
- `StrategyDebateFacts`
- `ReviewPositionFact`

### 5.2 AI 输出原则

AI 输出必须区分：
- facts used
- interpretation
- recommendation
- counterarguments
- uncertainty / confidence

markdown 只是展示形式，结构化 payload 才是 artifact 的核心。

当前代码落点：
- `ReviewAgentOutput`
- `StrategyProposalOutput`
- `ChallengerOutput`
- `JudgeOutput`

## 6. Transport / Orchestration 层

职责：
- 暴露 API / CLI / MCP / typed tools
- 提供本地 automation 入口
- 与外部 runtime 对接

对应模块：
- `apps/api`
- `mcp`
- `admin/cli.py`
- `scheduler`

这层的边界是：
- 可以把内核产物暴露出去
- 不能自己变成 accounting engine
- 不能在 transport 层重算 portfolio truth

## 关键契约：`fund-manager` 与 OpenClaw

从高阶方案角度看，两个系统之间应该只有三种稳定契约。

### 契约 A：Fact Pack

`fund-manager` -> `OpenClaw`

内容：
- deterministic facts
- deterministic decisions
- deterministic signals

用途：
- review
- debate
- channel summary
- AI augmentation

### 契约 B：AI Artifact

`OpenClaw` 或 in-repo AI runtime -> `fund-manager`

内容：
- structured review output
- strategy proposal
- challenge output
- judge output

要求：
- typed
- append-only
- 带 model / prompt / run metadata

### 契约 C：Execution Feedback

human / `OpenClaw` -> `fund-manager`

内容：
- `decision_run_id`
- `action_index`
- `feedback_status`
- note / created_by / feedback_date

要求：
- 由 `fund-manager` 写入 canonical audit artifact
- 由 `fund-manager` 执行 reconciliation

## 关键数据流

### 流程 1：导入与记账

1. 用户导入 holdings / transactions
2. 数据接入层做归一化
3. Canonical Truth 层更新 master data 与 ledger
4. 如为 transaction import，则同步 `position_lot`

### 流程 2：Daily Snapshot

1. 调度或人工触发 daily snapshot
2. 数据接入层刷新当前持仓基金资料与 NAV
3. Canonical Truth 层保存 `portfolio_snapshot`

### 流程 3：Daily Decision

1. 读取 canonical snapshot + active policy
2. Deterministic Decision 层生成 `decision_run`
3. 外部 runtime 可解释结果，但不能替换结果

### 流程 4：Weekly Review

1. `fund-manager` 组装 `WeeklyReviewFacts`
2. runtime 根据协议返回 `ReviewAgentOutput`
3. `fund-manager` 保存 `review_report`
4. markdown exporter 将 typed artifact 投影成可读报告

### 流程 5：Monthly Strategy Debate

1. `fund-manager` 组装 `StrategyDebateFacts`
2. runtime 完成 propose / challenge / judge
3. `fund-manager` 保存 `strategy_proposal` 和 debate log

### 流程 6：Watchlist / Radar

1. `fund-manager` 基于 deterministic rule 生成 signal outputs
2. 输出给 API / CLI / MCP / workflow context
3. AI 可以引用这些 signals，但不能把它们写回 canonical truth

## 模块职责重述

### `core/domain`

应该只放：
- 纯值逻辑
- 指标公式
- 与 infrastructure 无关的 domain rules

### `core/services`

应该只放：
- canonical truth 组装
- deterministic decision
- deterministic signal generation
- reconciliation

不应该放：
- prompt 文本
- channel 文案
- provider-specific orchestration

### `core/fact_packs.py`

应该只放：
- AI workflow 的 deterministic inputs
- 不依赖 runtime 实现的稳定 contract

### `core/ai_artifacts.py`

应该只放：
- AI workflow 的结构化 outputs
- 与 markdown projection 分离的 artifact schema

### `agents/workflows`

应该负责：
- workflow 级 orchestration
- persistence of artifacts
- run metadata

不应该负责：
- 自己重算 domain truth
- 自己定义 accounting logic

### `agents/runtime`

当前可保留：
- protocol
- prompt metadata helpers
- manual / lightweight runtime stubs

长期推荐：
- 将真实的模型路由和外部增强逐步外移到 `OpenClaw`

## 重要设计决策

### 1. 先有 truth，再有 AI

任何新能力都先判断：
- 是不是 canonical truth
- 是不是 deterministic decision
- 是不是 research signal
- 是不是 AI artifact

不要把这些对象混在一个 workflow 里一起说成“智能结果”。

### 2. research signal 也必须 deterministic

观察池、风向标和候选适配分析不是 prompt 推荐，而是 deterministic signal layer。

### 3. AI 产物必须结构化

报告和策略文本可以是 markdown，但必须有结构化 payload 作背书。

### 4. `OpenClaw` 是外部编排层，不是领域真相层

`OpenClaw` 可以决定：
- 什么时候跑
- 用哪个模型
- 怎么给人讲

但不能决定：
- 组合里真实持有什么
- policy 是否命中
- 某笔交易是否已经发生

## 推荐目标结构

在不要求立即重命名目录的前提下，推荐把代码逻辑理解为：

- `core/domain`: truth formula layer
- `core/services`: truth + deterministic decision layer
- `core/watchlist`: deterministic signal layer
- `core/fact_packs.py`: AI input contract layer
- `core/ai_artifacts.py`: AI output contract layer
- `agents/workflows`: artifact workflow layer
- `agents/runtime`: in-repo AI adapter layer
- `apps/api` / `mcp` / `admin` / `scheduler`: transport layer

如果后续继续重构，优先方向是：
- 保持 `fund-manager` 作为 domain kernel
- 弱化 in-repo AI runtime 的中心地位
- 强化 Fact Pack 与 AI Artifact 的 typed contract

## 下一步建议

下一步不是继续堆功能，而是继续做两类硬化：

1. 结构硬化
   - 让 truth / decision / signal / narrative / feedback 五类对象在代码里也更清楚

2. 契约硬化
   - 让 API / CLI / MCP / workflow 对这些对象的表达更加一致
