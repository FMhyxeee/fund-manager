# 基金投资管理器高阶方案（High-Level Solution）

## 文档定位

本文档描述当前仓库的高阶架构、模块边界和关键数据流。

如果你要扩展系统，先看：

1. [`AGENTS.md`](../AGENTS.md)
2. 本文档
3. [`doc/03-技术文档.md`](./03-%E6%8A%80%E6%9C%AF%E6%96%87%E6%A1%A3.md)

## 架构总览

当前实现可以分成 6 层：

1. **数据接入层**
   - CSV 导入器
   - AKShare 适配器
2. **事实与计算层**
   - 组合快照、收益、回撤等确定性服务
   - 基金日更同步服务
3. **持久化层**
   - SQLAlchemy 模型
   - Repository
   - Alembic 迁移
4. **工作流层**
   - 周报 workflow
   - 策略辩论 workflow
5. **运行入口层**
   - FastAPI
   - Scheduler CLI
   - 脚本
6. **Agent / 外部消费层**
   - Agent runtime
   - MCP server

## 当前模块职责

### 数据接入层

职责：

- 导入持仓 CSV
- 导入交易 CSV
- 获取基金公开资料
- 获取基金净值历史
- 做字段标准化

对应模块：

- `data_adapters/import_holdings.py`
- `data_adapters/import_transactions.py`
- `data_adapters/akshare_adapter.py`

### 事实与计算层

职责：

- 汇总当前持仓状态
- 生成组合快照
- 计算日 / 周 / 月收益、区间收益、回撤
- 做当前持仓基金的每日数据同步

对应模块：

- `core/domain/metrics.py`
- `core/services/analytics_service.py`
- `core/services/portfolio_service.py`
- `core/services/fund_data_sync_service.py`

### 持久化层

职责：

- 保存基金主数据
- 保存交易和持仓 lot
- 保存净值快照和组合快照
- 保存周报、策略提案、Agent 日志、系统事件

对应模块：

- `storage/models.py`
- `storage/repo/*.py`
- `storage/migrations/`

### 工作流层

职责：

- 将结构化事实交给 Agent
- 记录运行过程和结果
- 把最终结果持久化

对应模块：

- `agents/workflows/weekly_review.py`
- `agents/workflows/strategy_debate.py`

### 运行入口层

职责：

- 对外提供 HTTP API
- 提供 Dashboard
- 提供定时任务 CLI
- 提供批量脚本入口

对应模块：

- `apps/api/`
- `scheduler/`
- `scripts/`

### Agent / 外部消费层

职责：

- 给 Agent 提供受控的结构化读接口
- 给 MCP 客户端提供只读数据和模拟能力

对应模块：

- `agents/tools/portfolio_tools.py`
- `mcp/service.py`
- `mcp/server.py`

## 关键数据流

### 1. 导入持仓 / 交易

1. 用户上传 CSV
2. 导入器做校验和标准化
3. Repository 创建或复用 `portfolio` / `fund_master`
4. 写入 `position_lot` 或 `transaction`

### 2. Daily 同步与快照

1. Scheduler 触发 `daily_snapshot`
2. `FundDataSyncService` 找出当前组合有效持仓基金
3. 调用 AKShare 适配器刷新基金资料和最新净值
4. 增量写入 `nav_snapshot`
5. `PortfolioService.save_portfolio_snapshot()` 保存当日组合快照

### 3. Weekly Review

1. Workflow 从确定性服务读取组合事实
2. 生成结构化 `WeeklyReviewFacts`
3. 调用 `ReviewAgent`
4. 渲染 markdown
5. 保存到 `review_report`

### 4. Monthly Strategy Debate

1. Workflow 构建 `StrategyDebateFacts`
2. `StrategyAgent` 给出方案
3. `ChallengerAgent` 质疑
4. `JudgeAgent` 裁决
5. 保存到 `strategy_proposal` 与 `agent_debate_log`

### 5. API / MCP 读取

1. API 或 MCP 调用服务层 / tools
2. 服务层返回结构化 DTO
3. HTTP 和 MCP 层负责序列化，不直接计算账本数字

## 重要设计决策

### 1. 账本和 Agent 分离

所有权威数字都在确定性服务里算，Agent 只解释和挑战这些结果。

### 2. 快照与日志 append-only

历史报告、策略、快照和事件都保留，不做隐式覆盖。

### 3. 脚本复用服务层

脚本只作为入口，不再复制持久化逻辑。
例如每日持仓报告脚本已经改为复用 `FundDataSyncService`。

### 4. MCP 是可选传输层

MCP 不是核心账本的一部分，而是给外部 Agent 或工具提供只读接口的边界层。

## 当前限制

- REST API 仍然轻量，不覆盖所有内部能力
- daily sync 当前只同步现有模型可承接的数据
- OpenClaw / 外部 runtime 集成不在核心仓库内强耦合实现

## 扩展建议

后续新增能力时，优先遵守下面的方向：

1. 新的外部数据抓取，先放到 `data_adapters`
2. 新的权威计算，先放到 `core/services`
3. 新的 Agent 能力，优先通过 `agents/tools` 或 `mcp`
4. 新的展示入口，避免直接越层读 ORM 并自行计算
