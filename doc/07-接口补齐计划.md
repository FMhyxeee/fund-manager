# 07-接口补齐计划

这份文档基于 [`06-边界与接口清单`](./06-%E8%BE%B9%E7%95%8C%E4%B8%8E%E6%8E%A5%E5%8F%A3%E6%B8%85%E5%8D%95.md)，把 `fund-manager` 接下来应该补齐的接口明确成一份可执行计划。

目标不是“所有能力都暴露到所有 transport”，而是让 `OpenClaw` 能通过稳定、可审计、低歧义的接口调用 `fund-manager`，同时保持边界清晰。

## 1. 规划原则

### 1.1 谁先补

优先补这些能力：

- `OpenClaw` 在实际闭环中马上要调用的
- 当前代码里已经有 deterministic service / workflow，但还没有稳定接口的
- 会影响人工执行闭环完整性的

不优先补这些能力：

- 只是内部 repo/helper 能做，但没有明确对外调用场景的
- 为了“接口对称”而对称暴露、但没有实际编排价值的
- 会让边界变糊、把数据库细节泄漏到 `OpenClaw` 的

### 1.2 哪种 transport 负责什么

- API: 默认的读写主接口，给 OpenClaw 和未来 UI/automation 用
- CLI: 本机自动化和 cron 首选，适合同机调用与排障
- MCP: 保持 read-mostly，优先承担只读查询和轻量模拟
- agent tools: 给 `fund-manager` 内部 agent/workflow 用，保持 typed、最小、可审计

默认策略：

- canonical read model 要有 API
- write path 要有 API 或 CLI
- MCP 不默认承担高副作用写操作
- tool 只暴露 workflow 真正需要的最小集合

## 3. 关键缺口

### 缺口 A: decision read/write surface 不完整

现在已经能：

- 运行 `daily_decision` workflow
- 往 `decision_run` 落库
- 对单条 `decision_run` 记 feedback

但还不能稳定地：

- 通过 API 触发 `daily_decision`
- 通过 API 读取单条 `decision_run` 详情
- 通过 API 列出某个组合最近的 decision runs
- 通过 API 读取某条 decision 的 feedback 列表
- 通过 API 让 `OpenClaw` 不靠查库就拿到 action 明细

这是当前最关键的接口缺口。

### 缺口 B: policy surface 基本没暴露

现在 policy 只能通过 repo/service 或手写脚本落库。

缺少：

- 读取 active policy 的 API
- append-only 创建 policy 的 API
- 对应 CLI
- 对应只读 MCP / tool 视图

没有这层，`OpenClaw` 很难可靠地解释“为什么今天给出这个决策”。

### 缺口 C: canonical read API 不对齐

MCP 已经有：

- metrics
- valuation history
- fund nav history

但 API 没有对应只读入口。

这会导致：

- OpenClaw 用 HTTP 侧接入时能力不完整
- 不同 transport 的读模型不一致
- 后续 UI 或 webhook 集成只能绕路

### 缺口 D: workflow trigger API 不完整

目前 API 只开放了 weekly review。

缺少：

- `daily_snapshot` 手动触发 API
- `daily_decision` 手动触发 API
- `monthly_strategy_debate` 手动触发 API

这会迫使 `OpenClaw` 走 scheduler CLI 或内部脚本，不够标准。

### 缺口 E: CLI 还偏“频率驱动”，不够“动作驱动”

当前 CLI 只有 scheduler 入口，适合“按频率跑 job”，不适合明确的领域动作。

缺少更清晰的命令，例如：

- `policy show`
- `policy create`
- `decision run`
- `decision show`
- `decision feedback`
- `snapshot run`
- `strategy run`

对于本机自动化和排障，这组动作型 CLI 会比频率型 CLI 更稳。

## 4. 分阶段计划

## Phase 0: 补齐 manual execution loop 主链路

目标：

- 让 `OpenClaw` 不查数据库也能完整跑通 “生成决策 -> 发给人 -> 记录反馈 -> 导入交易后对账”。

### P0.1 API: decision 触发和读取

新增：

- `POST /api/v1/workflows/daily-decision/run`
- `GET /api/v1/decisions/{decision_run_id}`
- `GET /api/v1/decisions`
  - 支持 `portfolio_id`
  - 支持 `decision_date`
  - 支持 `limit`
- `GET /api/v1/decisions/{decision_run_id}/feedback`

响应目标：

- 返回完整结构化 decision payload
- 不只给 `decision_run_id`，还要给 `summary`、`final_decision`、`confidence_score`、`actions`
- 对 feedback 列表返回 append-only 历史和 linked transaction ids

原因：

- 这是 `OpenClaw` 最需要的最小闭环接口
- 当前只有 feedback 写接口，没有 decision 读接口，半边是空的

### P0.2 API: policy 读写

新增：

- `GET /api/v1/policies/active`
  - 支持 `portfolio_id`
  - 支持 `as_of_date`
- `POST /api/v1/policies`

写入规则：

- append-only
- 不做 in-place update
- 明确 `effective_from`
- fund target 使用 typed request schema

原因：

- `daily_decision` 的输入真相之一就是 active policy
- 没有 policy 接口，OpenClaw 无法稳定解释或初始化决策链路

### P0.3 API: workflow trigger 补齐

新增：

- `POST /api/v1/workflows/daily-snapshot/run`
- `POST /api/v1/workflows/monthly-strategy-debate/run`

原因：

- 触发面要对齐
- 不要让 `weekly_review` 成为唯一一个“有 API”的 workflow

### P0.4 CLI: 引入动作型命令

新增一个更明确的 domain CLI，优先命名成单独入口，例如：

- `fund-manager-admin policy show`
- `fund-manager-admin policy create`
- `fund-manager-admin decision run`
- `fund-manager-admin decision show`
- `fund-manager-admin decision feedback`
- `fund-manager-admin workflow run daily-snapshot`
- `fund-manager-admin workflow run daily-decision`
- `fund-manager-admin workflow run weekly-review`
- `fund-manager-admin workflow run monthly-strategy-debate`

当前状态：

- 已完成
- 新入口已落地为 `fund-manager-admin`
- 已实现 `policy show/create`、`decision list/show/run/feedback`、`workflow run daily-snapshot|daily-decision|weekly-review|monthly-strategy-debate`

如果不想新增入口，也可以先扩展现有 CLI，但命令语义必须从“频率”转成“动作”。

原因：

- `OpenClaw` 的 cron 和同机 automation 更适合调用动作型 CLI
- 排障时也更直观

## Phase 1: 补齐 canonical read model 的跨 transport 一致性

目标：

- 让 API、MCP、tool 至少在读模型上基本一致。

### P1.1 API 对齐 MCP 的只读能力

新增：

- `GET /api/v1/portfolios/{portfolio_id}/positions`
- `GET /api/v1/portfolios/{portfolio_id}/metrics`
- `GET /api/v1/portfolios/{portfolio_id}/valuation-history`
- `GET /api/v1/funds/{fund_code}/nav-history`

原因：

- 这些读能力已经在 MCP 存在，说明领域模型已经足够稳定
- API 不应该明显落后于 MCP

### P1.2 API 补 detail endpoints

新增：

- `GET /api/v1/reports/{report_id}`
- `GET /api/v1/strategy-proposals/{proposal_id}`

当前状态：

- 已完成第一批 detail endpoints
- 已新增 `GET /api/v1/reports/{report_id}`
- 已新增 `GET /api/v1/strategy-proposals/{proposal_id}`
- 已新增 `GET /api/v1/portfolios/{portfolio_id}/latest-report`
- 已新增 `GET /api/v1/portfolios/{portfolio_id}/latest-strategy-proposal`
- 已顺手修正 `GET /api/v1/reports` 在非空场景下的返回契约，使其不再依赖不存在的 `status` 列

原因：

- 现在只有 list，没有 detail
- `OpenClaw` 想转述具体内容时还得走别的路径

### P1.3 MCP 增补 decision / policy 只读接口

新增 read-only MCP tools：

- `portfolio_active_policy`
- `decision_run_get`
- `decision_run_list`
- `decision_feedback_list`
- `review_report_get`

当前状态：

- 已完成
- 新增只读 MCP tools：`portfolio_active_policy`、`decision_run_list`、`decision_run_get`、`decision_feedback_list`、`review_report_get`

原则：

- 只补只读，不把 MCP 变成高副作用写通道

### P1.4 Agent tools 最小补齐

新增 tool：

- `get_portfolio_metrics`
- `get_portfolio_valuation_history`
- `get_active_policy`
- `run_daily_decision`
- `get_decision_run`

当前状态：

- 已完成第一批最小补齐
- 新增 tool：`get_portfolio_metrics`、`get_portfolio_valuation_history`、`get_active_policy`、`run_daily_decision`、`get_decision_run`
- 已补充 side-effect tool：`record_decision_feedback`

是否补写操作 tool：

- 已完成
- 继续保持显式 side-effect 标注，并要求 `decision_run_id + action_index`

## Phase 2: 硬化接口契约，方便 OpenClaw 长期编排

目标：

- 让接口更适合长期自动化和跨 transport 调用，而不是只够本地试用。

### P2.1 统一响应 DTO

要求：

- 同一个领域对象在 API / MCP / tool 尽量复用同一套 DTO 语义
- 字段名统一
- 时间、Decimal、enum 的序列化规则统一

优先统一这些对象：

- portfolio summary
- portfolio snapshot
- portfolio metrics
- policy
- decision run
- decision feedback

### P2.2 统一审计字段

对所有 write / workflow 接口统一支持：

- `trigger_source`
- `created_by`
- `run_id`
- `idempotency_key` 或等价去重语义

原因：

- 方便 `OpenClaw` 做重复调用保护和 traceability

当前状态：

- 已完成第一批落地
- workflow trigger API 现在统一接受 `trigger_source`、`created_by`、`run_id`、`idempotency_key`
- `policy create` 现在统一接受 `created_by`、`run_id`、`idempotency_key`
- `idempotency_key` 当前通过稳定 `run_id` 派生实现等价去重语义
- 对 `daily_snapshot`、`daily_decision`、`weekly_review`、`monthly_strategy_debate`、`policy create` 已增加 duplicate `run_id` 冲突保护
- 现阶段是“稳定冲突保护”，不是“透明重放已有结果”；这已经足够支撑 OpenClaw 编排侧的重复调用保护

### P2.3 统一错误模型

目标：

- `portfolio not found`
- `decision run not found`
- `action index invalid`
- `policy missing`
- `snapshot incomplete due to missing NAV`

这些都应该有稳定的错误码或稳定 detail shape。

原因：

- `OpenClaw` 要根据错误类型决定下一步动作，而不是只看字符串

当前状态：

- API 现在统一返回稳定错误 envelope：
  - `detail`
  - `error.code`
  - `error.message`
  - `error.details`
- 已覆盖的稳定错误码包括：
  - `portfolio_not_found`
  - `decision_run_not_found`
  - `active_policy_not_found`
  - `fund_not_found`
  - `duplicate_run_id`
  - `action_index_invalid`
  - `invalid_period_range`
  - `validation_error`

### P2.4 明确 transport 分工并冻结边界

最终希望形成：

- API: canonical read + canonical write 的主入口
- CLI: 本机 automation 和调试入口
- MCP: read-mostly 查询入口
- tools: `fund-manager` 内部 workflow 的最小调用面

不建议走向：

- 同一个高副作用动作在四种 transport 上都全部暴露
- MCP 变成主要写入口
- repo/model 被直接暴露给 OpenClaw

## 5. 推荐交付顺序

建议按下面顺序做，而不是并行铺开：

1. `decision` 读写 API
2. `policy` 读写 API
3. `daily_decision` / `daily_snapshot` / `monthly_strategy_debate` 的 workflow trigger API
4. 动作型 CLI
5. API 与 MCP 的 read parity
6. MCP decision/policy 只读补齐
7. DTO / error / idempotency 硬化

## 6. 建议的第一批 issue 切分

可以直接切成下面几张票：

1. `Decision API read surface`
   - `GET /decisions`
   - `GET /decisions/{id}`
   - `GET /decisions/{id}/feedback`

2. `Decision workflow trigger API`
   - `POST /workflows/daily-decision/run`

3. `Policy API`
   - `GET /policies/active`
   - `POST /policies`

4. `Workflow trigger parity`
   - `POST /workflows/daily-snapshot/run`
   - `POST /workflows/monthly-strategy-debate/run`

5. `Canonical read API parity`
   - positions
   - metrics
   - valuation history
   - fund nav history

6. `Action-oriented CLI`
   - policy
   - decision
   - workflow

## 7. 完成标准

当下面几件事成立时，可以认为“接口补齐第一阶段完成”：

- `OpenClaw` 可以不查库，只通过稳定接口完成 daily decision 闭环
- `OpenClaw` 可以读到 active policy 并解释 decision 来源
- 主要 workflow 都有标准触发入口，而不是只靠内部 service/脚本
- API 不再明显落后于 MCP 的核心 read model
- CLI 不再只会按频率跑，而能按明确领域动作跑

当前判断：

- Phase 1 已完成
- 下一阶段应转入 P2，重点做 DTO 语义统一、错误模型统一，以及 `run_id / created_by / trigger_source / idempotency` 的契约硬化
