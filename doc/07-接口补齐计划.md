# 07-接口补齐计划

本文档基于当前代码状态，整理 `fund-manager` 还应该如何继续补齐接口。

目标不是“所有能力都暴露到所有 transport”，而是让：

- `OpenClaw` 可以稳定调用
- 人工排障可以直接操作
- transport 之间的语义尽量一致

## 1. 当前接口格局

### API

当前已经承担：

- canonical read
- canonical write
- workflow trigger
- decision / policy / report / strategy proposal 读取

推荐继续作为主接口。

### CLI

当前已经分成：

- `fund-manager-scheduler`
  - 频率驱动
- `fund-manager-admin`
  - 动作驱动

推荐继续作为同机 automation 和 debug 的主入口。

### MCP

当前定位：

- read-mostly
- 轻量外部 agent 查询入口

推荐继续保持保守，不把高副作用写路径默认暴露到 MCP。

### Internal typed tools

当前定位：

- 给 `fund-manager` 内部 workflow 调用
- 保持最小、typed、可审计

## 2. 当前接口能力里已经比较稳定的部分

### 已经稳定的主链路

- portfolio snapshot / positions / metrics / valuation history
- fund profile / NAV history
- active policy read / append-only create
- daily snapshot trigger
- daily decision trigger
- weekly review trigger
- monthly strategy debate trigger
- decision list / detail / feedback
- report detail / strategy proposal detail

### 已经形成的 typed contract

- Fact Pack
  - `core/fact_packs.py`
- AI Artifact
  - `core/ai_artifacts.py`

这意味着：

- transport 层可以围绕稳定 DTO 展开
- workflow 结果解释不再依赖 runtime 实现文件

## 3. 下一阶段最值得补齐的方向

## Phase A：Signal layer 对外接口

当前 watchlist / leader / fit 已经有 deterministic service 雏形，但对外 transport 还不统一。

推荐补齐：

### API

- `GET /api/v1/watchlist/candidates`
- `GET /api/v1/watchlist/style-leaders`
- `GET /api/v1/watchlist/fit`

### CLI

- `fund-manager-admin watchlist candidates`
- `fund-manager-admin watchlist leaders`
- `fund-manager-admin watchlist fit`

### MCP

- `watchlist_candidates`
- `watchlist_style_leaders`
- `watchlist_candidate_fit`

原则：

- signal 输出是 deterministic read model
- 不是 policy 写路径
- 不是 execution command

## Phase B：DTO 与错误模型进一步统一

虽然主要接口已经存在，但还可以继续硬化：

### DTO 一致性

优先统一这些对象在 API / CLI / MCP 中的字段表达：

- portfolio summary
- portfolio snapshot
- policy
- decision run
- decision feedback
- signal output

### 错误模型一致性

继续保持和补齐稳定错误码：

- `portfolio_not_found`
- `decision_run_not_found`
- `active_policy_not_found`
- `action_index_invalid`
- `duplicate_run_id`
- `signal_seed_missing`（未来如需要）

## Phase C：Workflow artifact 读取面

当前 AI workflow 已经有 typed artifact contract，但外部读取面还可以更清楚。

推荐方向：

- 让 weekly review detail 更明确区分 facts vs narrative
- 让 strategy proposal detail 更明确区分 facts / proposal / challenge / judgment
- 如果未来增加 signal snapshot，也保持 append-only 读取面

## 4. 接口设计原则

1. API 是主入口，CLI 是同机操作入口，MCP 保持 read-mostly。
2. 同一个领域对象在不同 transport 上尽量复用相同语义。
3. transport 不拥有 canonical truth。
4. transport 不重新发明 Fact Pack 或 AI Artifact。
5. transport 不应暴露 raw ORM model。

## 5. 不应该再走的方向

- 为了“接口对称”而把所有写动作都暴露到 MCP
- 在 route handler 或 CLI handler 里重算 portfolio truth
- 让 `OpenClaw` 依赖查库而不是稳定接口
- 在 transport 层重新拼一套 review / strategy artifact schema

## 6. 推荐交付顺序

1. signal layer API
2. signal layer CLI
3. signal layer MCP
4. artifact detail DTO 统一
5. 剩余 error / idempotency / response envelope 硬化

## 7. 完成标准

当下面几件事成立时，可以认为接口层已经进入“长期可编排”状态：

1. `OpenClaw` 不查数据库，也能完成 daily decision 与 weekly/monthly artifact 读取。
2. signal layer 有稳定的 API / CLI / MCP 表达。
3. Fact Pack 和 AI Artifact 在 transport 层不再被重新定义。
4. API / CLI / MCP 的对象语义基本一致。
5. transport 层只是桥，不再承载领域真相。
