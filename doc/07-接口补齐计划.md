# 07-接口使用指南与补齐计划

本文档记录当前精简核心的接口状态，以及下一步值得补齐的接口硬化工作。

旧版计划中的 daily decision、weekly review、strategy proposal、MCP、scheduler、admin CLI 已经不属于当前产品面，不再作为补齐目标。

## 1. 当前接口格局

### FastAPI

当前主接口。面向 OpenClaw、脚本和外部调用者。

公开 route 只保留：

- health
- portfolios
- funds
- transactions
- watchlist

详细 endpoint 和示例见 [`../skills/fund-manager-interfaces/references/core-api.md`](../skills/fund-manager-interfaces/references/core-api.md)。

### Core services

面向仓库内部调用和测试。

稳定入口：

- `PortfolioService.assemble_portfolio_snapshot`
- `TransactionService.list_transactions`
- `TransactionService.get_transaction`
- `TransactionService.append_transaction`
- `FundWatchlistService.list_items`
- `FundWatchlistService.add_item`
- `FundWatchlistService.remove_item`

### Storage repositories

面向 service 实现和测试 fixture。不要把 repository 当成外部接口。

### Direct SQL

只用于：

- 诊断
- migration
- 明确的 repair workflow

普通集成和 OpenClaw 编排不应直接写数据库。

## 2. 当前已稳定的能力

### 组合读取

- 组合列表
- snapshot
- positions
- metrics
- valuation history
- 缺失 NAV 显式暴露

### 基金读取

- fund profile
- NAV history

### 交易账本

- transaction list
- transaction detail
- controlled append
- append 后 deterministic lot sync

### 观察池

- active list
- include removed list
- add / reactivate
- soft remove
- 与 accounting truth 分离

### 错误响应

API 错误使用稳定 envelope：

```json
{
  "detail": "Portfolio not found",
  "error": {
    "code": "portfolio_not_found",
    "message": "Portfolio not found",
    "details": null
  }
}
```

调用方应依赖 `error.code`，不要解析自然语言错误文本。

## 3. 下一阶段补齐方向

### Phase A：文档和技能同步

状态：已完成。

目标：

- `README.md` 指向接口 skill。
- `AGENTS.md` 记录接口 skill。
- `doc/06` 和本文件与精简核心一致。
- `skills/fund-manager-interfaces` 成为 agent 使用接口时的入口。

### Phase B：交易追加幂等性

当前 `POST /transactions` 可接收 `external_reference`、`source_name`、`source_reference`，但数据库层没有强制幂等约束。

推荐补齐：

- 明确 external reference 的唯一性策略。
- 增加 repository/service 层重复提交保护。
- 返回稳定冲突错误码，例如 `duplicate_transaction_reference`。
- 增加 API 和 service 测试。

### Phase C：API DTO 和字段命名硬化

推荐继续检查：

- Decimal 字段是否在 API 文档中足够清楚。
- `portfolio_id` 与 `portfolio_name` 的解析优先级是否需要显式写入 OpenAPI description。
- `lot_sync` 响应是否需要稳定 Pydantic model，而不是 `dict[str, object]`。
- watchlist 的 `style_tags` 是否需要长度和字符约束。

### Phase D：OpenClaw 调用适配

在 OpenClaw 侧更新所有调用点：

- 禁止调用已删除的 MCP、scheduler、admin CLI、workflow route。
- 交易写入统一走 `POST /api/v1/transactions`。
- 组合、基金、观察池读取统一走 FastAPI。
- 根据 `error.code` 做分支处理。

### Phase E：只读诊断脚本

如果需要保留本机排障效率，可以新增只读脚本，但不要恢复旧 admin CLI。

推荐范围：

- show current alembic version
- count core tables
- check missing NAV for active positions
- inspect latest transaction lot sync state

这些脚本应保持只读，写 repair 逻辑必须单独命名并带测试或操作说明。

## 4. 不应该补回的方向

- MCP ledger write tools
- scheduler jobs
- AI report / strategy / decision workflows
- policy persistence
- dashboard UI
- CSV import API
- broker/order integrations
- 在 route handler 中直接写 accounting tables

## 5. 完成标准

接口层进入稳定状态的标准：

1. OpenClaw 不查数据库即可完成组合读取、交易追加和观察池管理。
2. 所有公开接口都有 tests 覆盖。
3. 所有写路径都走 service。
4. 交易追加具备幂等或重复提交保护。
5. 文档、skill、README 和 AGENTS 对当前接口的描述一致。
