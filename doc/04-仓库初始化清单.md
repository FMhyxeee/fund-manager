# 仓库初始化清单

## 文档定位

这份文档不再描述“一个空仓库该怎么从 0 开始搭建”，而是记录：

- bootstrap 阶段原本想完成什么
- 当前哪些关键里程碑已经落地
- 新环境第一次拉起时建议按什么顺序走

当前真实运行方式以 `README.md` 为准。

## 仍然值得继续补强的方向

这些不再属于 bootstrap 阻塞项，但仍是合理的后续增强：

- 补齐 signal layer 的对外 API / MCP / CLI
- 统一更多 DTO 和错误模型
- 收敛无关 lint / type debt
- 硬化脚本与服务层边界
- 继续把生产级 runtime 能力外移到 `OpenClaw`

## 新环境首次拉起顺序

1. 安装依赖

```bash
uv sync --extra dev --extra data
```

2. 执行迁移

```bash
uv run alembic upgrade head
```

3. 选择性导入基金基础数据

```bash
uv run python scripts/import_all_funds.py
```

4. 导入持仓或交易

- 通过 API
- 通过 `fund-manager-admin`
- 或在测试环境中直接调用 importer

5. 跑 workflow 验证主链路

```bash
uv run fund-manager-admin workflow run daily-snapshot --portfolio-id 1 --as-of-date 2026-04-15
uv run fund-manager-admin workflow run daily-decision --portfolio-id 1 --decision-date 2026-04-15
uv run fund-manager-admin workflow run weekly-review --portfolio-id 1 --period-end 2026-04-15
uv run fund-manager-admin workflow run monthly-strategy-debate --portfolio-id 1 --period-end 2026-04-15
```

## 当前应如何理解“初始化完成”

现在这个仓库的“初始化完成”不只是指：

- 目录建好了
- 数据库起得来

还应包括：

- deterministic truth layer 已形成
- deterministic decision layer 已形成
- typed fact / artifact contract 已分层
- runtime 不再混入 canonical truth

也就是说，bootstrap 阶段已经结束，当前仓库处于“可持续演进”的阶段。

## 经验总结

目前已经被验证正确的工程方向：

1. 先做确定性账本，再做 AI 叙事。
2. 先做 typed contract，再做 runtime 细节。
3. workflow 先组装 facts，再调 AI，再 append-only 持久化。
4. 脚本只能是入口，不能复制核心写路径。
5. 文档必须跟着当前代码结构一起更新，否则很快失真。
