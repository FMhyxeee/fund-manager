# 仓库初始化清单（Repository Bootstrap Checklist）v1

## 1. 文档目的

本文档用于指导项目从 0 到可开发状态的初始化过程。

适用对象：
- 你自己
- Codex
- 未来参与该仓库的其他 Agent / 模型

目标：
- 在最短路径内完成仓库搭建
- 为后续 Agent 与 loop 工作流打好基础
- 避免一开始就把架构做乱

---

## 2. 初始化目标

完成本文档后，仓库应具备以下最小能力：

1. 项目目录结构已创建
2. `AGENTS.md` 已放入仓库根目录
3. 基础 Python 项目已初始化
4. 数据库模型与迁移机制已就位
5. 持仓导入脚本可运行
6. 基金公开数据适配器可运行
7. 组合基础指标可计算
8. weekly loop 具备最小可执行骨架
9. 基础测试框架可运行

## 4. 仓库初始化步骤

## Step 1. 创建仓库与基础文件

### 目标
建立最小可维护仓库。

### 动作
- 初始化 Git 仓库
- 创建主目录结构
- 添加 `README.md`
- 添加 `AGENTS.md`
- 添加 `.gitignore`
- 添加 `.env.example`

### 检查点
- 根目录下可以看到 `AGENTS.md`
- `docs/` 中已有核心文档
- 空目录至少通过 `.gitkeep` 保留

---

## Step 2. 初始化 Python 工程

### 推荐
使用 `uv` 或 `poetry` 均可，优先 `uv`。

### 最低依赖建议
- `fastapi`
- `sqlalchemy`
- `alembic`
- `pydantic`
- `pydantic-settings`
- `pandas`
- `akshare`
- `httpx`
- `pytest`
- `pytest-asyncio`
- `python-dotenv`

### 可选依赖
- `rich`
- `typer`
- `jinja2`
- `polars`
- `orjson`

### 检查点
- `pyproject.toml` 可安装依赖
- `pytest` 能空跑通过
- `python -m` 方式能执行模块

---

## Step 3. 建立配置系统

### 目标
建立统一配置入口，避免后续散落读取环境变量。

### 建议文件
- `core/config.py`

### 配置项建议
- `APP_ENV`
- `DATABASE_URL`
- `OPENCLAW_BASE_URL`
- `OPENCLAW_WORKSPACE`
- `OPENAI_PROVIDER_ENABLED`
- `GLM_PROVIDER_ENABLED`
- `REPORT_OUTPUT_DIR`
- `DEFAULT_PORTFOLIO_NAME`

### 检查点
- `.env.example` 可清晰说明所有变量
- 配置读取失败时给出明确错误

---

## Step 4. 设计数据库与迁移机制

### 首批表
必须先建：
- `fund_master`
- `portfolio`
- `transaction`
- `nav_snapshot`
- `portfolio_snapshot`
- `review_report`
- `strategy_proposal`
- `agent_debate_log`

### 建议文件
- `storage/models.py`
- `storage/db.py`
- `storage/repo/*.py`
- `storage/migrations/`

### 检查点
- 能生成并执行第一版迁移
- 本地数据库可以创建全部核心表
- repository 层可独立调用

---

## Step 5. 编写导入器（最优先）

### 目标
你需要先把自己的持仓导进去。

### 先支持的导入内容
- 初始持仓 CSV
- 后续交易流水 CSV

### 最小 CSV 格式建议

#### 持仓导入
```csv
fund_code,fund_name,units,avg_cost,total_cost,portfolio_name
000001,示例基金A,1000,1.235,1235,main
```

#### 交易流水导入
```csv
trade_date,fund_code,trade_type,units,amount,fee,nav_at_trade,source,note
2026-03-01,000001,buy,100,123.5,0,1.235,manual,initial buy
```

### 必做校验
- 基金代码不能为空
- 份额不能为负
- 金额和份额至少一项有意义
- 日期格式必须规范
- 未识别交易类型必须报错

### 检查点
- 可成功导入一份测试 CSV
- 导入失败时有明确错误位置
- 导入不会 silently skip 无效行

---

## Step 6. 接入基金公开数据适配器

### v1 数据源建议
首版只接 AKShare。

### 最小能力
- 根据基金代码获取基本信息
- 获取基金历史净值
- 获取基金公开资料摘要

### 建议文件
- `data_adapters/akshare_adapter.py`
- `data_adapters/normalize_fund_data.py`

### 适配器规则
- 统一输出内部 DTO / schema
- 外部字段名不要直接污染领域层
- 所有日期与数值格式要统一

### 检查点
- 给定基金代码能返回标准化结果
- 拉不到数据时能给出明确异常
- 同一个基金数据格式在整个系统内一致

---

## Step 7. 实现确定性指标服务

### 目标
这是系统核心，必须在 Agent 之前完成。

### 首批必须实现的函数
- `compute_current_value`
- `compute_unrealized_pnl`
- `compute_weight`
- `compute_period_return`
- `compute_max_drawdown`
- `build_portfolio_snapshot`

### 建议文件
- `core/domain/metrics.py`
- `core/services/portfolio_service.py`
- `core/services/analytics_service.py`

### 检查点
- 关键函数有单元测试
- 多基金组合可正确计算权重与收益
- 缺失净值时处理逻辑明确

---

## Step 8. 构建 weekly loop 最小骨架

### 目标
先做最重要的 weekly loop，不要一开始就上 daily / monthly 全套。

### 最小流程
1. 读取当前组合
2. 拉过去一周的净值数据
3. 构建组合快照
4. 生成结构化 metrics
5. 调用 `ReviewAgent`
6. 保存 `review_report`

### 当前阶段要求
- 可以先不接多 Agent
- 可以先由单个 Agent 输出周报
- 先把输入和输出格式跑通

### 检查点
- 能手动触发一次 weekly loop
- 数据、报告、日志能落盘或入库

---

## Step 9. 增加多 Agent 工作流

### 上线顺序建议
1. `ReviewAgent`
2. `StrategyAgent`
3. `ChallengerAgent`
4. `JudgeAgent`

### 强约束
- `StrategyAgent` 不能直接成为最终结论
- `JudgeAgent` 必须看到 challenge 结果
- debate 结果必须可落库

### 检查点
- 一次策略讨论至少包含 proposal / challenge / final judgment 三段内容
- 模型输出结构稳定，可解析

---

## Step 10. 接入 OpenClaw runtime

### 目标
把系统从“手动脚本”升级到“可持续调用的 agent runtime”。

### 建议接入位
- `agents/runtime/openclaw_bridge.py`

### 最小功能
- 提交 workflow 请求
- 传入上下文与工具集合
- 获取 agent 输出
- 记录 run id / trace ref

### 检查点
- 可以通过统一入口调用一次 agent workflow
- trace/ref 可以保存到数据库

---

## Step 11. 加入调度器

### v1 建议
先实现：
- 手动命令触发 weekly loop
- 再加入 cron / daemon 定时触发

### 建议文件
- `scheduler/jobs.py`
- `scheduler/trigger_loop.py`

### 检查点
- 支持手动执行
- 支持定时执行
- 失败时能重试或至少能记录错误

---

## Step 12. 建立基础测试与 fixture

### 必备 fixture
- 单基金持仓样本
- 多基金组合样本
- 历史净值样本
- 异常交易样本
- 导入 CSV 样本

### 最先写的测试
- metrics 单元测试
- import pipeline 集成测试
- weekly loop happy path 测试

### 检查点
- CI 或本地一条命令能跑核心测试
- 核心财务逻辑具备回归保护

---

## 5. 建议的开发里程碑

## Milestone 1：账本内核完成
完成标志：
- 持仓导入成功
- 净值可更新
- 组合快照可生成
- 核心 metrics 测试通过

## Milestone 2：周报完成
完成标志：
- weekly loop 可手动运行
- `ReviewAgent` 可输出周报
- 报告可保存

## Milestone 3：多 Agent 策略讨论完成
完成标志：
- `StrategyAgent`、`ChallengerAgent`、`JudgeAgent` 可串起来
- 策略建议与反驳可留痕

## Milestone 4：定时 loop 完成
完成标志：
- weekly loop 可定时执行
- 失败会记录并可复查

---

## 6. 仓库初始化当天的任务单

建议你在第一天只做这些：

1. 建仓库
2. 放入文档与 `AGENTS.md`
3. 初始化 Python 工程
4. 建第一版数据库模型
5. 做持仓 CSV 导入
6. 写 AKShare adapter
7. 写 metrics 单测
8. 写一个手动 weekly loop 脚本

不要在第一天做：
- 前端页面
- 多模型辩论
- 复杂 Prompt 打磨
- 太多 provider 接入
- 自动调仓逻辑

---

## 7. Codex 启动任务建议

当你把这些文件放进仓库后，可以先让 Codex 从以下任务开始：

1. 创建目录结构与空文件
2. 初始化 `pyproject.toml`
3. 创建 SQLAlchemy 模型与 Alembic 初始迁移
4. 实现 `import_holdings.py`
5. 实现 `akshare_adapter.py`
6. 实现 `metrics.py`
7. 为 `metrics.py` 写单元测试
8. 编写 `weekly_review.py` 的最小骨架

---

## 8. 完成定义

仓库初始化完成，不是指“项目做完了”，而是指满足以下条件：

- 代码结构清晰
- 数据模型成立
- 导入与计算链条打通
- Agent 有最小接入点
- 可以开始持续开发

如果以上 5 条都满足，就可以正式进入功能开发阶段。
