# 基金投资管理器项目蓝图（Blueprint）v1

## 1. 文档目标

本文档用于定义项目的**产品边界、系统定位、核心能力、阶段目标与成功标准**。

适用对象：
- 项目发起人（你）
- Codex / 其他 coding agent
- 未来的协作模型与自动化流程

本文档回答的问题：
- 这个项目到底要做什么
- 不做什么
- 为什么要这样设计
- 第一阶段应该先做哪些能力
- 如何判断 v1 是否成功

---

## 2. 项目定位

该项目是一个：

**面向个人使用的、自托管的、由 Agent 驱动的基金投资复盘与策略辅助系统。**

它不是：
- 自动交易系统
- 券商 / 支付平台接管系统
- 高频量化系统
- 面向公众开放的 SaaS 平台
- 完全依赖大模型进行账务计算的黑盒系统

该项目的核心价值是：
- 管理自己的基金组合
- 统一持仓、交易、净值、收益、回撤等事实数据
- 通过 Agent 定时复盘组合表现
- 让不同模型围绕策略进行讨论和挑战
- 生成长期可追踪、可审计的投资记录与策略建议

---

## 3. 产品愿景

构建一个“个人基金组合操作系统”，使其具备以下能力：

1. **统一账本**
   - 能准确表达你的基金持仓、交易流水、成本与收益

2. **持续复盘**
   - 每日 / 每周 / 每月自动生成组合复盘

3. **策略讨论**
   - 多个 Agent / 多个模型围绕组合状态给出建议、反驳与裁决

4. **长期记忆**
   - 保存你的风险偏好、投资原则、禁忌、历史判断与事后复盘

5. **决策支持**
   - 给你提供可执行的操作建议，但不自动下单

---

## 4. 核心设计原则

### 4.1 账本计算必须确定性

以下内容必须由确定性代码完成，不允许直接交给 LLM 计算：
- 持仓份额
- 成本价
- 总投入
- 当前市值
- 累计收益
- 区间收益
- 最大回撤
- 仓位占比
- 再平衡偏离

### 4.2 Agent 负责分析，不负责篡改账本

Agent 只负责：
- 解释
- 复盘
- 质疑
- 比较
- 生成草案
- 形成结论

Agent 不负责：
- 直接写 SQL
- 修改历史账本
- 直接下单
- 跳过工具调用直接输出“确定事实”

### 4.3 决策支持优先，自动执行后置

v1 不做：
- 自动买卖
- 支付宝自动操作
- 自动扣款
- 交易型闭环

v1 只做：
- 数据统一
- 指标计算
- 复盘
- 策略建议
- 风险提示

### 4.4 多 Agent 要有分工，不做无约束群聊

不同 Agent 必须有明确职责：
- 有的负责事实
- 有的负责复盘
- 有的负责提出策略
- 有的负责反驳
- 有的负责裁决

### 4.5 所有结论必须可追溯

任何复盘结论或策略建议都必须能追溯到：
- 哪次 loop 产生
- 哪个 Agent 产生
- 调用了哪些工具
- 使用了哪些输入数据
- 最终是否被采纳

---

## 5. 目标用户

当前仅有一个目标用户：

**你自己。**

因此 v1 的产品特征应该是：
- 单用户
- 自托管
- 低运维复杂度
- 可随时调整
- 优先服务个人使用效率，而非通用性

---

## 6. 关键使用场景

### 场景 1：初始化组合

用户导入当前基金持仓，系统建立第一版组合账本。

### 场景 2：每日更新

系统定时获取基金公开净值，刷新组合快照，生成简短日报。

### 场景 3：每周复盘

系统自动汇总过去一周的收益、回撤、仓位变化、风险提示，并产出周报。

### 场景 4：策略讨论

多个 Agent 围绕当前组合状态讨论：
- 是否继续定投
- 是否降低某只基金权重
- 是否加入观察名单
- 是否需要月度再平衡

### 场景 5：事后复盘

用户回看历史报告，比较“当时建议”和“后来结果”，校正系统策略质量。

---

## 7. v1 产品范围

### 7.1 必做能力

#### A. 持仓管理
- 初始化导入基金持仓
- 查看当前持仓
- 查看持仓权重
- 查看单基金成本与盈亏

#### B. 交易管理
- 导入或录入买入 / 卖出 / 分红 / 转换流水
- 基于交易重建持仓状态

#### C. 基金详情
- 基金代码、名称、类型、基金经理、风险等级等基础资料
- 历史净值 / 近期表现

#### D. 组合指标
- 当前总市值
- 总投入
- 累计收益
- 日 / 周 / 月收益
- 最大回撤
- 仓位占比

#### E. 自动复盘
- Daily loop
- Weekly loop
- Monthly loop

#### F. 多 Agent 策略讨论
- Strategy Agent 生成建议
- Challenger Agent 反驳
- Judge Agent 给出最终结论

#### G. 历史留痕
- 保存报告
- 保存策略草案
- 保存 debate log
- 保存事件日志

### 7.2 暂不纳入 v1 的能力

- 自动交易执行
- 自动读取支付宝个人私有持仓
- 多用户权限系统
- 原生移动端 App
- 高频策略
- 大规模资产类别扩展（股票、债券、期权等）
- 复杂的组合优化器
- 全自动的宏观新闻驱动系统

---

## 8. 非功能性目标

### 8.1 可维护性
- 领域逻辑与模型 Prompt 分离
- 数据接入与账本计算解耦
- 工具层与 Agent 层解耦

### 8.2 可测试性
- 账本和指标计算必须有单元测试
- 导入流程必须有集成测试
- 核心 loop 必须可回放

### 8.3 可审计性
- 每次 loop 有 run id
- 每个 Agent 输出有日志
- 每份策略建议可回溯

### 8.4 可扩展性
- 未来可支持更多数据源
- 未来可支持更多模型
- 未来可引入再平衡策略模板

---

## 9. 业务主线

系统的核心业务主线为：

**持仓与交易事实 → 净值更新 → 指标计算 → Agent 分析 → 策略讨论 → 报告落库 → 用户决策**

其本质不是“聊天”，而是“围绕组合状态进行周期性研究与决策支持”。

---

## 10. v1 成功标准

当以下条件成立时，可视为 v1 成功：

### 10.1 数据层面
- 可以稳定导入初始持仓
- 可以导入交易流水
- 可以更新基金净值
- 可以生成组合快照

### 10.2 计算层面
- 能准确计算单基金盈亏
- 能准确计算组合总收益
- 能计算回撤与仓位占比

### 10.3 Agent 层面
- Daily / Weekly / Monthly loop 能正常运行
- 多 Agent 能完成一轮策略讨论
- 能保存完整讨论过程

### 10.4 产品层面
- 你能定期查看复盘报告
- 你能基于系统建议做手动决策
- 你能回看过去建议是否靠谱

## 11. 里程碑规划

### Milestone 1：账本可用
目标：
- 持仓导入
- 基金信息获取
- 净值更新
- 基础收益计算

### Milestone 2：周报可用
目标：
- Weekly loop
- 周报自动生成
- 报告落库

### Milestone 3：多 Agent 讨论可用
目标：
- Strategy / Challenger / Judge 工作流跑通
- 策略建议可追溯

### Milestone 4：长期复盘闭环
目标：
- 月报
- 长期策略建议
- 历史效果回看

## 12. 风险与边界

### 12.1 数据源风险
公开基金数据接口可能变动，系统需要抽象适配层。

### 12.2 认证风险
订阅登录模式适合个人自托管，但不适合作为正式多用户生产体系。

### 12.3 策略幻觉风险
Agent 的结论不应被视为事实，必须依赖工具和数据证据。

### 12.4 过度自动化风险
系统提供建议，不直接执行交易。
