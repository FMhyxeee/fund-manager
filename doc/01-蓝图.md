# 基金投资管理器蓝图（Blueprint）

> 当前状态（2026-04-18）：本文保留为历史蓝图背景。当前可执行产品面已经精简为 ledger / portfolio read model / fund read / watchlist core。涉及 AI workflow、policy/decision persistence、MCP、scheduler、admin CLI、report/dashboard 的内容不再代表当前实现；当前接口以 `AGENTS.md`、`README.md`、`doc/06-边界与接口清单.md`、`doc/07-接口补齐计划.md` 和 `skills/fund-manager-interfaces/` 为准。

## 文档定位

本文档定义 `fund-manager` 的产品定位、系统边界和长期演进方向。

它回答四个问题：
- `fund-manager` 到底是什么
- AI 在这个项目里到底扮演什么角色
- 当前代码已经把哪些边界落地了
- 后续演进应该继续围绕什么内核展开

如果蓝图与当前实现冲突，以代码和 `README.md` 为准；但后续改造应尽量向本文档收敛。

## 一句话定义

`fund-manager` 是一个面向个人、自托管的**投资领域内核**。

它负责维护组合的 canonical truth，执行确定性会计与 policy 评估，保存 append-only 的决策与复盘产物，并通过稳定接口把这些结构化事实提供给人类操作者与外部 AI runtime。

AI 不是这个系统的“真相来源”，而是这个系统之上的**可替换分析层**。

## 它是什么

`fund-manager` 是：
- 个人投资组合的领域真相系统
- 一个以确定性代码为核心的 portfolio accounting / review / policy engine
- 一个带审计留痕的人工执行闭环系统
- 一个为外部 runtime、agent、UI 提供稳定结构化接口的领域服务

## 它不是什么

`fund-manager` 不是：
- 自动交易系统
- 券商接管系统
- 靠 prompt 直接算账的 AI 黑盒
- 依赖某个固定模型才能成立的“AI 产品壳”
- 面向公网和多租户的 SaaS 平台

## 产品核心价值

这个项目要解决的，不是“让 AI 给出更多观点”，而是先解决“什么才是组合里的真相”。

它的核心价值分成四层：

### 1. 投资真相集中化

把以下对象统一到一个可信内核中：
- 持仓
- 交易
- 基金资料
- 净值历史
- 组合快照
- policy
- 决策记录
- 人工反馈
- 交易回挂关系

### 2. 确定性决策能力

基于 canonical facts 和 policy 规则，稳定地产生：
- 组合估值
- 收益率与回撤
- band breach
- rebalance gap
- deterministic daily decision

### 3. 研究与观察能力

在不污染 canonical truth 的前提下，提供 deterministic 的研究读模型：
- watchlist candidates
- style leaders
- candidate vs portfolio fit
- 后续可扩展的 market radar / signal snapshot

### 4. AI 叙事与讨论能力

让 AI 基于结构化 facts 做：
- 周报解释
- 策略草案
- 反方挑战
- 裁决总结

但这些都只是 narrative artifact，不是 canonical truth。

## 系统稳定性的定义

如果一个系统想长期稳定，必须先满足以下条件：

### 1. 无 AI 也能正确运行

即使完全关闭 LLM，这些流程仍应成立：
- holdings import
- transaction import
- NAV refresh
- portfolio snapshot
- policy evaluation
- daily decision run
- feedback recording
- reconciliation

### 2. 真相不依赖对话上下文

系统中的 canonical truth 不能依赖：
- 某次聊天
- 某个 prompt
- 某个 agent 的一段自然语言输出

### 3. 历史必须可审计

过去某天的：
- 组合状态
- 决策结论
- 周报内容
- 策略判断
- 人工执行反馈

都必须能够追溯回：
- 哪次 workflow
- 哪些结构化输入
- 哪个 run_id
- 哪个 artifact

### 4. AI 可以替换，领域内核不能漂移

未来即使：
- 从 OpenAI 换到别的 provider
- 从本地 agent runtime 换到 OpenClaw
- prompt 模板完全重写

`fund-manager` 的 canonical truth 和 deterministic 决策也不能因此改变。

## 对象模型：五类东西必须拆开

这是本项目后续演进时最重要的边界。

### 1. Canonical Facts

由 `fund-manager` 拥有。

包括：
- `portfolio`
- `fund_master`
- `transaction`
- `position_lot`
- `nav_snapshot`
- `portfolio_snapshot`
- 确定性 metrics

要求：
- 只能由 deterministic code 产生或归一化
- 不允许 prompt-only 逻辑定义

### 2. Deterministic Decisions

由 `fund-manager` 拥有。

包括：
- `decision_run`
- policy band 检查结果
- suggested rebalance amount

要求：
- 只能基于 canonical facts + policy 规则产生
- AI 可以解释，但不能替代

### 3. Research Signals

由 `fund-manager` 的 deterministic read-model 拥有。

包括：
- watchlist
- style leaders
- candidate fit
- 后续 market radar

要求：
- 不是 canonical accounting truth
- 不是 policy truth
- 不是自动执行指令

### 4. AI Narratives

由 AI runtime 生成，`fund-manager` 可选择持久化。

包括：
- weekly review narrative
- strategy proposal
- challenger output
- judge output

要求：
- 必须区分 facts / interpretation / recommendation / uncertainty
- 只能作为 artifact 持久化，不能回写成真相

### 5. Human Execution Feedback

由 `fund-manager` 拥有。

包括：
- `decision_feedback`
- `decision_transaction_link`

要求：
- 必须引用具体 deterministic action
- 只能由人工显式触发

## AI 的正确位置

AI 在这个项目里应该被定义成三件事：
- 解释器
- 假设生成器
- 争论者

AI 不应该被定义成：
- 账本
- policy engine
- 执行器
- canonical source of truth

因此，项目设计上应该始终遵守：

1. 先有 deterministic fact pack，再有 AI narrative。
2. 先有 domain service，再有 prompt。
3. 先有 typed artifact，再有 markdown 展示。

## 当前代码里的落点

这套蓝图不再只是抽象约束，当前仓库已经有一批明确落点：

### 内核与 typed contract

- `src/fund_manager/core/fact_packs.py`
  - `WeeklyReviewFacts`
  - `StrategyDebateFacts`
  - `ReviewPositionFact`
- `src/fund_manager/core/ai_artifacts.py`
  - `ReviewAgentOutput`
  - `StrategyProposalOutput`
  - `ChallengerOutput`
  - `JudgeOutput`

### AI runtime bridge

- `src/fund_manager/agents/runtime/contracts.py`
  - runtime protocol
- `src/fund_manager/agents/runtime/shared.py`
  - prompt metadata loading and tiny shared helpers
- `src/fund_manager/agents/runtime/review_agent.py`
- `src/fund_manager/agents/runtime/strategy_agent.py`
- `src/fund_manager/agents/runtime/challenger_agent.py`
- `src/fund_manager/agents/runtime/judge_agent.py`
  - manual runtime implementations only

### Workflow 与渲染

- `src/fund_manager/agents/workflows/weekly_review.py`
  - 组装 deterministic facts，调用 runtime，保存 review artifact
- `src/fund_manager/agents/workflows/strategy_debate.py`
  - 组装 deterministic facts，串联 propose / challenge / judge
- `src/fund_manager/reports/exporters/markdown_exporter.py`
  - 把 typed artifacts 投影成 markdown

当前代码已经明确表达：
- facts 不再由 runtime 模块拥有
- AI outputs 不再由 runtime 模块定义为 canonical contract
- runtime 更接近桥接层，而不是项目中心

## 与 OpenClaw 的边界

`fund-manager` 负责：
- 投资领域真相
- 确定性会计与决策
- 研究信号读模型
- append-only 审计产物
- 对外 typed interface

`OpenClaw` 负责：
- agent runtime
- model/profile 选择
- prompt orchestration
- channel delivery
- 外部搜索与跨项目工具
- cron、重试、通知策略

简化理解：
- “是什么”归 `fund-manager`
- “怎么讲”归 AI / OpenClaw
- “何时跑、发给谁”归 OpenClaw

## 当前版本的推荐理解

截至当前阶段，项目的正确理解应从：

> 一个 agent-driven fund portfolio review and strategy assistant

调整为：

> 一个以投资领域真相和确定性决策为核心、以 AI 作为可替换分析层的个人投资内核系统

这不是措辞优化，而是系统设计原则的调整。

## 未来演进方向

接下来合理的演进顺序应该是：

1. 继续硬化 canonical truth 与 deterministic decision
2. 把 research signal layer 独立清楚，并补齐 API / CLI / MCP 表达
3. 继续把 AI runtime 与 domain kernel 的契约稳定下来
4. 再扩展 weekly / monthly narrative 和 multi-agent debate
5. 最后扩展展示层与交互层
