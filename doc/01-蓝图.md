# 基金投资管理器项目蓝图（Blueprint）

## 文档定位

本文档描述项目的产品目标、范围边界，以及“当前版本已经做到哪里”。

说明：

- `README.md` 是上手文档和当前实现入口
- 本文档是产品蓝图，不是逐行代码规范
- 如果蓝图和代码实现冲突，以代码和 `README.md` 为准

## 项目定位

`fund-manager` 是一个：

**面向个人使用、自托管、以确定性账本为核心、由 Agent 驱动的基金组合复盘与策略辅助系统。**

它不是：

- 自动交易系统
- 券商接管系统
- 面向公众的 SaaS 平台
- 依赖 LLM 直接计算账本的黑盒系统

## 产品价值

系统要解决的核心问题是：

1. 把基金持仓、交易、净值和组合表现统一到一个可信账本里。
2. 持续生成每日、每周、每月的复盘结果。
3. 让 Agent 基于结构化事实做解释、挑战和策略讨论。
4. 保存历史报告和策略判断，便于后续回看与复盘。

## v1 范围

### 已经落地的主线能力

- 基金主数据、交易、持仓 lot、净值快照、组合快照等核心持久化模型
- 持仓 CSV / 交易 CSV 导入
- AKShare 基金资料与净值历史适配器
- 当前持仓基金的每日数据同步
- 组合层面的确定性统计和快照
- 每周复盘工作流
- 每月策略辩论工作流
- REST API、Dashboard、可选 MCP 只读服务

### 当前仍未纳入的能力

- 自动下单或自动执行交易
- 自动接管支付宝 / 券商私有账户
- 多用户权限系统
- 面向外部用户的稳定 SaaS API
- 完整市场行情仓库

## 核心原则

### 1. 确定性账本优先

以下内容必须由确定性代码完成：

- 持仓份额
- 成本价
- 当前市值
- 浮动盈亏
- 组合权重
- 日 / 周 / 月收益
- 区间收益
- 最大回撤

### 2. Agent 负责分析，不负责改账

Agent 可以：

- 解释事实
- 组织复盘
- 提策略草案
- 提反驳意见
- 形成最终建议

Agent 不可以：

- 直接写 SQL 改账
- 隐式修正历史交易
- 跳过工具调用直接输出“权威数字”

### 3. 每个结论都要可追溯

报告和策略结论应当能追溯到：

- 哪次 workflow 产生
- 哪个 agent 产生
- 使用了哪些结构化输入
- 结果被保存到了哪里

## 关键使用场景

### 初始化组合

导入持仓或交易流水，建立组合账本。

### 每日更新

同步当前持仓基金最新资料和净值，保存组合快照，生成日报。

### 每周复盘

基于结构化事实生成周报，记录组合表现和关注点。

### 每月策略讨论

基于当前组合状态和近期表现，让多个 Agent 对配置调整提出和挑战建议。

### 事后复盘

回看历史报告、历史策略和后续结果，对判断质量做追踪。

## 当前版本的边界说明

截至当前实现：

- “每日更新”已经进入正式 scheduler 链路
- “每周复盘”和“每月策略辩论”已经有完整 workflow
- HTTP API 仍偏轻量，部分底层能力只通过 scheduler、脚本或 MCP 暴露
- 文档和设计均默认单用户、自托管、低运维复杂度

## 后续优先项

当前蓝图下，比较合理的下一批增强方向是：

1. 补齐基金搜索和净值历史的 REST API
2. 暴露 monthly strategy debate 的 HTTP 触发入口
3. 丰富 daily sync 的数据范围和观测指标
4. 为报告和策略结果增加更好的展示页面
