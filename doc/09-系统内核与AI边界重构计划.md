# 系统内核与 AI 边界重构计划

> 当前状态（2026-04-18）：本文是精简核心前的历史重构计划。当前仓库已经执行更激进的简化：AI/runtime/workflow/policy/decision/MCP/scheduler/admin/report/dashboard 已从产品面移除。当前后续计划以 `doc/07-接口补齐计划.md` 为准。

## 1. 目标

这份计划把前面的文档边界，转成可执行的代码重构路线。

目标不是“继续加 AI 功能”，而是把项目从：

> 以 agent / workflow 为中心的投资辅助系统

重构为：

> 以投资领域内核为中心、以 AI 作为可替换分析层的稳定系统

## 2. 目标状态

重构完成后，系统应满足以下条件：

### 2.1 内核稳定

即使没有 AI，以下流程仍然全部可运行：
- import
- sync
- snapshot
- policy evaluation
- daily decision
- feedback recording
- reconciliation

### 2.2 对象分类清晰

代码和接口层都明确区分：
- canonical facts
- deterministic decisions
- research signals
- AI narratives
- human execution feedback

### 2.3 契约稳定

`OpenClaw` 或其他 runtime 与 `fund-manager` 的交互，必须围绕稳定契约进行：
- Fact Pack
- AI Artifact
- Execution Feedback

### 2.4 历史可追踪

任何 persisted artifact 都应能回溯到：
- run_id
- workflow_name
- trigger_source
- facts source
- model / prompt metadata（若为 AI artifact）

## 3. 重构总原则

1. 先改定义，再改代码。
2. 先拆对象，再拆模块。
3. 先稳定 deterministic core，再调整 AI runtime。
4. 先引入 typed contract，再做更大规模的目录迁移。
5. 不为了“AI 更方便”牺牲 truth 的边界。

## 4. 当前阶段状态

截至当前版本，已经完成的部分如下：

### 已完成

- 文档层已经完成从“AI 驱动 assistant”到“投资领域内核”的重新定义
- `AGENTS.md` 已把 truth / decision / signal / artifact / runtime bridge 的边界写清楚
- `core/fact_packs.py` 已落地
- `core/ai_artifacts.py` 已落地
- `agents/runtime/contracts.py` 已落地
- `agents/runtime/shared.py` 已落地
- manual runtime 已拆到 `review_agent.py`、`strategy_agent.py`、`challenger_agent.py`、`judge_agent.py`
- weekly review / strategy debate 已改成依赖 typed facts 和 typed artifacts，而不是 runtime 私有 DTO

### 部分完成

- AI workflow 的 typed contract 已成型，但更广的 transport 表达还未完全统一
- signal layer 已有 service 雏形，但对外接口与 provenance 还不完整
- runtime bridge 已明显收缩，但生产级 runtime 仍需继续外移到 `OpenClaw`

### 尚未完成

- daily decision 的解释性 artifact 还没有独立成统一模式
- watchlist / leader / fit 的 API / CLI / MCP 表达还需要系统补齐
- 一些旧的技术债仍与本轮边界重构并行存在，例如类型债、个别服务逻辑问题、脚本层写入边界问题

## 5. 分阶段计划

### Phase 0：定义和契约重构

目标：
- 让项目文档、代码评审标准和后续 issue 切分围绕新的系统模型展开

状态：
- 已完成

结果：
- `AGENTS.md`
- `doc/01-蓝图.md`
- `doc/02-高阶方案.md`
- `doc/08-观察池与风向标MVP设计.md`
- 本计划文档

完成标准：
- 项目正式定义从“AI 驱动 assistant”切换为“投资领域内核”
- watchlist / radar 被明确归入 research signal layer

### Phase 1：代码对象模型重构

目标：
- 在代码层引入 truth / decision / signal / narrative / feedback 的对象边界

状态：
- 已大体完成

已落地动作：

#### 1. Fact Pack DTO

当前已落地：
- `WeeklyReviewFacts`
- `StrategyDebateFacts`
- `ReviewPositionFact`

所在文件：
- `src/fund_manager/core/fact_packs.py`

#### 2. AI Artifact DTO

当前已落地：
- `ReviewAgentOutput`
- `StrategyProposalOutput`
- `StrategyAction`
- `ChallengerOutput`
- `JudgeOutput`

所在文件：
- `src/fund_manager/core/ai_artifacts.py`

#### 3. Runtime bridge 收缩

当前已落地：
- `src/fund_manager/agents/runtime/contracts.py`
- `src/fund_manager/agents/runtime/shared.py`
- 四个 manual runtime implementation 文件

阶段结论：
- AI workflow 的输入输出已经不再混在 runtime 私有模块或 generic dict 中
- “研究信号”和“确定性决策”在 DTO 层面已开始可区分

### Phase 2：Workflow 责任重构

目标：
- workflow 只做 orchestration，不再隐式承担 truth 定义职责

状态：
- 部分完成

已落地动作：
- weekly review workflow 使用 `WeeklyReviewFacts`
- strategy debate workflow 使用 `StrategyDebateFacts`
- markdown exporter 使用 core-level typed artifacts，而不是 runtime 私有定义

下一步动作：

#### 1. Daily Decision 纯 deterministic 化

要求：
- `DailyDecisionWorkflow` 只围绕 deterministic decision run 工作
- AI 解释如果存在，也作为额外 artifact，而不是 `decision_run` 的一部分

#### 2. Weekly Review 双层化继续硬化

要求：
- `review_report` 背后始终有结构化 payload
- markdown 只是渲染结果

#### 3. Strategy Debate 双层化继续硬化

要求：
- `strategy_proposal` 保存时，facts 与 AI judgment 分层清楚

完成标准：
- 每个 workflow 都能指出哪些步骤是 deterministic，哪些步骤是 AI
- AI 步骤失败时，不影响 canonical truth 的正确性

### Phase 3：Signal Layer 硬化

目标：
- 把观察池与风向标从“附加功能”升级为正式的 deterministic signal layer

状态：
- 已有基础，尚未完成

当前落点：
- `src/fund_manager/core/watchlist/service.py`

下一步动作：

#### 1. 统一 signal 命名与接口

围绕以下能力整理接口：
- watchlist candidates
- style leaders
- candidate fit

#### 2. 视情况增加 signal snapshot

如果后续需要做：
- 历史对比
- 周度变化
- 研究回放

可以新增 append-only 的 signal snapshot artifact，而不是把信号混进 canonical snapshot。

#### 3. 为 signal layer 增加 provenance

例如：
- `seed_version`
- `rule_version`
- `as_of_date`

完成标准：
- watchlist / leader / fit 都被视为 deterministic signals
- 它们能稳定进入 review / strategy context，但不污染 decision truth

### Phase 4：AI Runtime 外移与桥接

目标：
- 让 `fund-manager` 不再把真实 AI runtime 视为内核一部分

状态：
- 部分完成

已落地动作：
- in-repo runtime 已拆成 protocol、shared helper、manual adapters

下一步动作：

#### 1. 保留 in-repo manual runtime

用途：
- 测试
- contract 验证
- 本地最小闭环

#### 2. 把生产级 runtime 视为外部桥接

由 `OpenClaw` 负责：
- model selection
- provider routing
- web search
- channel interaction
- retry and fallback

`fund-manager` 负责：
- 提供 Fact Pack
- 接收 AI Artifact
- 做 artifact persistence

#### 3. 明确桥接接口

后续可考虑显式定义：
- `generate_weekly_review_narrative(fact_pack) -> artifact`
- `run_strategy_debate(fact_pack) -> artifact set`

完成标准：
- 移除“必须依赖 in-repo runtime 才能工作”的认知
- `fund-manager` 成为稳定的 domain kernel

### Phase 5：Transport 契约统一

目标：
- API / CLI / MCP / tool 都围绕统一对象表达

状态：
- 部分完成。MCP 已覆盖 portfolio、decision、report、watchlist、fund、model simulation 等 read-mostly 查询，并新增 OpenClaw 可调用的受控 `transaction_append` 写入口。

建议动作：
- API 优先作为 canonical read/write 的主接口
- CLI 作为本地 automation / debug 入口
- MCP 保持 read-mostly；例外只允许明确、受控、append-only 的领域写操作，例如记录人类确认后的交易流水
- internal tools 只暴露 workflow 真正需要的最小集合

完成标准：
- 同一对象在多种 transport 上的语义尽量一致
- transport 层不再自行拼接 domain truth
- 外部 runtime 不需要直接连接数据库即可查询交易、记录人工确认交易，并获得 lot 重建和 decision reconciliation 的确定性结果

## 6. 建议的后续代码改造清单

下面是更贴近代码库的落地点。

### 6.1 `core/services`

继续作为 deterministic kernel 的中心，负责：
- truth assembly
- decision evaluation
- signal generation
- reconciliation

建议：
- 继续把 AI 相关逻辑隔离在 core 外侧

### 6.2 `core/watchlist`

当前可以视为 signal layer 雏形。

建议：
- 继续留在 deterministic service 范畴
- 后续若增长明显，可演化为 `core/signals`

### 6.3 `agents/workflows`

建议保留，但职责收缩为：
- 组装 fact pack
- 调用 AI adapter
- 保存 append-only artifact

### 6.4 `agents/runtime`

建议继续定位为：
- in-repo AI adapter / stub layer

而不是：
- 项目的根中心

### 6.5 `storage/models`

建议在未来演化时更明确地区分：
- truth tables
- decision audit tables
- narrative artifact tables
- optional signal artifact tables

## 7. 验收标准

当以下条件成立时，可以认为这次重构真正落地：

1. 项目定义已经从“AI 驱动”改成“领域内核驱动”
2. 代码中 truth / decision / signal / narrative / feedback 的对象边界更清楚
3. daily decision 完全 deterministic
4. weekly / monthly AI 产物有结构化 contract
5. watchlist / radar 被稳定归类为 signal layer
6. `OpenClaw` 不再需要越过 `fund-manager` 直接接触数据库或重算事实

## 8. 当前最值得做的下一步

基于当前状态，最合理的下一批动作是：

1. 把 signal layer 的 API / CLI / MCP 接口系统补齐
2. 给 watchlist outputs 增加更明确的 signal metadata 与 provenance
3. 继续审视 daily decision，把解释性 artifact 与 `decision_run` 彻底分层
4. 统一 workflow artifact 在 transport 层的表达方式
5. 在修边界的同时，逐步清理并行存在的类型债和脚本层边界问题
